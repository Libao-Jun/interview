## 导航守卫

| 类型         | 守卫                                                          |
| ------------ | ------------------------------------------------------------- |
| 全局前置守卫 | `beforeEach`                                                  |
| 全局解析守卫 | `beforeResolve`                                               |
| 全局后置守卫 | `afterEach`                                                   |
| 路由独享     | `beforeEnter`                                                 |
| 组件内       | `beforeRouteEnter` / `beforeRouteUpdate` / `beforeRouteLeave` |

## 路由传参

- `path`+ `query` 传参: 在浏览器地址栏中显示参数，类似`get`传参
- `name` + `params` 传参: 在浏览器地址栏中不显示参数，类似`post`传参

<Callout type="warn" title="注意事项">
  - 刷新：不会丢失 `query` 里面的数据 - 刷新：会丢失 `params` 里面的数据
</Callout>

## $route 和 $router

| 维度         | `$route`                              | `$router`                        |
| ------------ | ------------------------------------- | -------------------------------- |
| **本质**     | 当前激活的路由**信息对象**            | 整个路由**实例**                 |
| **属性**     | `path`/`params`/`query`/`name`/`meta` | `push`/`replace`/`go`/`addRoute` |
| **职责**     | 只读，获取当前路由状态                | 读写，控制路由跳转               |
| **是否可变** | 每次导航产生新对象                    | 全局唯一实例                     |

## 动态路由+路由权限

📚 常见的三种实现方案

| 方案                          | 核心思路                                                          | 适用场景             |
| ----------------------------- | ----------------------------------------------------------------- | -------------------- |
| **① 后端返回完整路由**        | 后端存路由配置，前端 `import.meta.glob` 映射组件，`addRoute` 注册 | 权限频繁变动的中后台 |
| **② 前端全量路由 + 权限过滤** | 前端定义全部路由 + `meta.roles`，后端返回角色标识过滤             | 角色固定/页面少      |
| **③ 公共路由 + 后端菜单**     | 公共路由内置在前端，业务路由后端返回动态追加                      | 多角色/公共页面统一  |

1️⃣ 方案一：后端返回完整路由

```
登录 → 获取 Token → 请求后端菜单接口 → 后端返回树形路由结构 → 通过 addRoute 动态注册路由
```

- **原理**：后端存储路由配置（path、component、meta），前端拿字符串路径通过 `import.meta.glob` 映射为组件
- **优点**：权限完全由后端控制，前端改动少，角色/菜单变更无需发版
- **缺点**：前后端需约定 component 命名规范，新增页面需后端同步配置
- **适用**：菜单结构复杂、权限变动频繁的中后台系统

2️⃣ 方案二：前端全量路由 + 后端返回权限标识过滤

```
登录 → 获取角色/权限标识 → 前端全量路由按 meta.roles 过滤 → 动态注册 → 渲染菜单
```

- **原理**：前端定义全部路由，每条路由标注 `meta.roles`，后端只返回用户拥有的权限/角色编码，前端按编码过滤
- **优点**：路由配置集中在前端，后端只需管角色分配；路由跳转逻辑更直观
- **缺点**：新增页面需前端编码+发版；权限粒度为角色级，颗粒度粗
- **适用**：角色固定、页面较少的管理系统

3️⃣ 方案三：公共路由 + 后端返回菜单路由

```
登录 → 初始化前端公共路由（内置） → 请求菜单接口 → 后端返回动态路由部分 → 通过 addRoute 动态注册路由
```

- **原理**：`createRouter` 时注册公共路由（静态），守卫中请求后端获取用户专属路由，追加注册
- **优点**：公共页面不受权限影响（如首页、404），仅业务路由动态
- **缺点**：与方案一类似，需后端同步路由配置
- **适用**：多角色但公共页面统一的系统

📙 方案对比

| 维度         | 后端路由               | 前端过滤                     | 混合模式            |
| ------------ | ---------------------- | ---------------------------- | ------------------- |
| 路由配置存放 | 后端数据库             | 前端代码                     | 公共前端 + 业务后端 |
| 菜单变更成本 | 改数据库即生效         | 前端改代码+发版              | 公共发版，业务改库  |
| 权限粒度     | 菜单+按钮级            | 路由+角色级                  | 菜单+按钮级         |
| 安全性       | 高（无权限路由不注册） | 中（路由仍注册，靠守卫拦截） | 高                  |
| 实施复杂度   | 中（需约定组件映射）   | 低                           | 中                  |

## hash vs history 模式

| 维度       | hash 模式           | history 模式             |
| ---------- | ------------------- | ------------------------ |
| URL 样式   | `http://xxx/#/path` | `http://xxx/path`        |
| 原理       | `hashchange` 事件   | `pushState/replaceState` |
| 服务器配置 | 无需                | 需配置（否则刷新 404）   |
| SEO        | 不利                | 友好                     |

## 切换路由实现保存草稿

### 解决方案

| 方案                                | 持久化      | 适用场景         | 复杂度 |
| ----------------------------------- | ----------- | ---------------- | ------ |
| `onBeforeRouteLeave` + localStorage | ✅          | 单页表单         | 低     |
| 全局导航守卫                        | ✅          | 多页面统一处理   | 中     |
| Pinia Store 管理                    | ✅          | 复杂业务，多草稿 | 中     |
| `keep-alive`                        | ❌ 刷新丢失 | 简单缓存         | 低     |
| 自定义对话框                        | ✅          | 需要良好交互     | 中     |

<Callout tittle="推荐">
中小项目用方案一或三，大型项目结合 Pinia + 全局守卫 + 自定义弹窗，体验最好。
</Callout>


### 方案详解

<Callout type="idea" title="方案一：onBeforeRouteLeave + localStorage">
利用 Vue Router 提供的 `onBeforeRouteLeave（离开组件触发）` 组合式 API，
在用户离开当前路由时拦截并提示，将表单数据存入 `localStorage`。
页面加载时从 `localStorage` 恢复数据。正式提交后清除草稿。

```sh 
# 实现步骤
1. 标记表单是否有修改，监听表单输入，标记为已修改
2. 保存草稿到 localStorage
3. 恢复草稿
4. 路由离开时拦截
5. 正式提交后清除草稿
```
</Callout>

<Callout type="idea" title="方案二：全局导航守卫 + 自定义事件（多页面统一拦截）">
在路由配置文件中使用全局的 `beforeEach` 守卫，
通过 `meta` 字段标记需要草稿保护的路由，
结合自定义事件或回调来触发保存逻辑。

```sh
# 实现步骤
1. 路由配置中添加 meta 标记
2. 全局守卫（beforeEach）拦截
3. 页面组件中监听事件，onUnmounted保存草稿，onMounted 恢复草稿
```
</Callout>

<Callout type="idea" title="方案三：Pinia Store 集中管理（推荐用于中大型项目）">
将草稿的状态、保存、恢复、清除逻辑全部封装在 Pinia Store 中，页面组件只负责调用 store 的方法，实现状态集中管理和逻辑复用。
```sh
# 实现步骤
1. 创建草稿 Store
2. 创建可复用的组合式函数
3. 在页面组件中使用
```
</Callout>

<Callout type="idea" title="方案四：keep-alive 组件缓存（无需持久化的快速方案）">
利用 Vue 内置的 `<keep-alive>` 组件缓存页面实例，切换路由时组件不会被销毁，表单数据保留在内存中。适用于不需要持久化存储的场景。
```sh
# 实现步骤
1. 路由视图使用 keep-alive，inlcude 需要缓存的组件名称列表
2. 页面组件需要定义 name
3. 动态管理缓存的组合式函数，组件激活时（从缓存恢复），组件失活时（被缓存）
```
</Callout>

<Callout type="idea" title="方案五：自定义对话框 + 后端存储（企业级完整方案）">
将草稿保存到后端，支持多端同步、版本管理，配合自定义弹窗组件提供良好的用户体验。适用于需要跨设备、团队协作的企业级应用。
```sh
# 实现步骤（流程：对话框组件 → API封装 → 组合式函数 → 页面使用）
1. 自定义确认对话框组件（保存草稿和放弃修改）
2. 后端 API 封装（保存草稿、获取草稿、删除草稿）
3. 增强版草稿组合式函数
4. 页面组件中使用
```
</Callout>
