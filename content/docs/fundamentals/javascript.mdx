## 数据类型

### 分类

| 基本类型（7）                         | 引用类型          |
| ------------------------------------- | ----------------- |
| `number` / `string` / `boolean`      | `Object`          |
| `undefined` / `null` / `symbol` / `bigint` | `Array` / `Function` / `Date` / `RegExp` 等 |

### 核心区别

- 基本类型：值存储在栈中，赋值是值拷贝
- 引用类型：引用存储在栈中，对象存储在堆中，赋值是引用拷贝

## 类型判断

1️⃣ 四种主要方式

| 方式                  | 基本类型 | null  | Array | NaN   | 自定义类 |
| --------------------- | -------- | ----- | ----- | ----- | -------- |
| `typeof`              | 准确    | `"object"` | `"object"` | `"number"` | —      |
| `instanceof`          | —       | —     | 准确  | —     | 准确     |
| `Array.isArray()`          | 数组      | —     | —  | —     | —     |
| `Object.prototype.toString.call()` | 准确 | 准确 | 准确 | 准确 | 准确 |

2️⃣ typeof — 判断基础类型

```js
typeof "hello"; // 'string'
typeof 42; // 'number'
typeof true; // 'boolean'
typeof undefined; // 'undefined'
typeof Symbol(); // 'symbol'
typeof 10n; // 'bigint'
typeof function () {}; // 'function'

// 两个坑
typeof null; // 'object' ← 历史遗留 bug
typeof []; // 'object' ← 无法区分数组和普通对象
typeof {}; // 'object'
```

3️⃣ Object.prototype.toString.call() — 精确类型

```js
Object.prototype.toString.call("hello"); // '[object String]'
Object.prototype.toString.call(42); // '[object Number]'
Object.prototype.toString.call(null); // '[object Null]'
Object.prototype.toString.call(undefined); // '[object Undefined]'
Object.prototype.toString.call([]); // '[object Array]'
Object.prototype.toString.call({}); // '[object Object]'
Object.prototype.toString.call(new Date()); // '[object Date]'
Object.prototype.toString.call(/abc/); // '[object RegExp]'
Object.prototype.toString.call(new Map()); // '[object Map]'
Object.prototype.toString.call(new Set()); // '[object Set]'
```

4️⃣ instanceof — 判断实例的构造函数（基于原型链）

```js
[] instanceof Array   // true
[] instanceof Object  // true（数组也是对象）
new Date() instanceof Date // true

// 局限：基础类型不行
'hello' instanceof String // false
42 instanceof Number      // false

// 局限：跨 iframe 不准确
// iframe 里的数组和当前窗口的 Array 不是同一个构造函数
```

5️⃣ Array.isArray() — 专门判断数组

```js
Array.isArray([]); // true
Array.isArray({}); // false
Array.isArray("hello"); // false
// 跨 iframe 也能正确判断
```
## JS中的错误类型

### 内置错误类型

<Callout title="JavaScript提供的错误对象">
JavaScript 提供了 `7` 种核心错误对象，它们都继承自 `Error`。
</Callout>

1️⃣ 7 种核心错误对象

- `Error` — 所有错误类型的通用基类，也可直接用于自定义抛错。
- `SyntaxError` — 语法错误，通常在`代码解析/编译阶段`抛出，无法被同作用域的 `try...catch` 捕获。
- `ReferenceError` — 引用错误，当访问一个`未声明的变量`时抛出。
- `TypeError` — 类型错误，当`值的类型不符合预期`时抛出，比如调用非函数或访问 null 的属性。
- `RangeError` — 范围错误，当值不在允许的集合或范围内时抛出。
- `URIError` — URI 处理错误，当 `decodeURIComponent` 等函数接收到格式错误的 URI 时抛出。
- `EvalError` — 历史遗留错误，原本在 `eval()` 使用不当时抛出。现代 JS 已不再主动抛出此错误，仅保留对象以保证兼容性。

2️⃣ 错误类型判断技巧

```js
try {
  // 可能出错的代码
} catch (err) {
  if (err instanceof TypeError) {
    // 处理类型错误
  } else if (err instanceof ReferenceError) {
    // 处理引用错误
  } else if (err.code === 403) {
    // 处理自定义业务错误
  } else {
    // 其他未知错误
  }
}
```

### 自定义错误类型
可以通过继承 `Error` 创建业务专属错误，这能保留原始错误信息和堆栈。

```js
class BusinessError extends Error {
  constructor(message, code) {
    super(message);
    this.name = "BusinessError";
    this.code = code;
  }
}

throw new BusinessError("用户权限不足", 403);
```

## 类型转换

### 隐式转换规则

- `==` 比较时会进行类型转换，`===` 不会
- 对象转基本类型：依次调用 `[Symbol.toPrimitive]` → `valueOf()` → `toString()`
- `+` 运算符：有字符串则拼接，否则数值相加
- `-` / `*` / `/` 运算符：全部转数值

### 经典题目

```js
[] == ![]     // true  → [] == false → '' == 0 → 0 == 0
[] == []      // false → 两个不同引用
'' == 0       // true
null == undefined // true
```

## 原型与原型链

### 核心关系

```
实例.__proto__ === 构造函数.prototype
构造函数.prototype.constructor === 构造函数
Object.prototype.__proto__ === null  // 链的终点
```

### 查找机制

当访问对象属性时，沿着 `__proto__` 逐级向上查找，直到 `null`。

```
实例 → 构造函数.prototype → Object.prototype → null
```

### 继承方案

| 方案               | 优点               | 缺点                       |
| ------------------ | ------------------ | -------------------------- |
| 原型链继承         | 简单               | 引用类型共享、无法传参     |
| 构造函数继承       | 可传参、不共享     | 无法继承原型方法           |
| 组合继承           | 兼顾两者           | 调用两次父构造函数         |
| 寄生组合继承       | 最优传统方案       | 写法稍复杂                 |
| ES6 `class extends`| 语法简洁           | 本质仍是原型链 + 寄生组合  |

## 内存溢出与内存泄漏

<Callout title="概念">
- **内存溢出**：内存不足无法分配（即程序申请的内存超过了系统能分配的上限，导致程序崩溃）
- **内存泄漏**：不需要的内存仍然被持有，无法被回收和释放，导致内存占用持续增加，最终导致程序崩溃。
</Callout>

## 进程与线程

1️⃣ 定义

- 进程：操作系统资源分配的最小单位
- 线程：CPU 调度的最小单位

2️⃣ 进程和线程的区别

| 维度       | 进程                           | 线程                             |
| ---------- | ------------------------------ | -------------------------------- |
| **定义**   | 操作系统资源分配的最小单位     | CPU 调度的最小单位               |
| **资源**   | 拥有独立的内存空间、文件句柄等 | 共享进程的内存空间和资源         |
| **开销**   | 创建销毁开销大，切换慢         | 创建销毁开销小，切换快           |
| **通信**   | 进程间通信复杂（共享内存）     | 线程间通过共享内存直接通信       |
| **隔离性** | 一个进程崩溃不影响其他进程     | 一个线程崩溃可能导致整个进程崩溃 |
| **独立性** | 进程之间相互独立               | 线程之间相互依赖                 |
| **关系**   | 容器（至少有一个主线程）       | 某个进程执行体                   |

## 防抖与节流

1️⃣ 定义

- 防抖：停止触发事件后`间隔指定时间`，然后执行回调（最后一次有效）。
- 节流：在被连续触发的事件中，`指定时间间隔内`只执行一次（第一次/最后一次）事件回调。

2️⃣ 应用场景

| 场景        | 用什么   | 原因             |
| ----------- | -------- | ---------------- |
| 搜索框输入  | `防抖`   | 等用户输入完     |
| 提交按钮    | `防抖`   | 防止重复提交     |
| 自动保存    | `防抖`   | 等编辑停止       |
| 窗口 resize | **节流**   | 等操作完成       |
| 滚动事件    | **节流** | 持续监控频率控制 |
| 鼠标移动    | **节流** | 高频更新需限频率 |
| 游戏射击    | **节流** | 控制技能冷却     |
| 埋点上报    | **节流** | 减少请求次数     |

## 闭包

<Callout title="闭包的定义">
通俗来说，就是一个函数（通常是内部函数）可以读取另一个函数（外部函数）中的局部变量。
</Callout>

📙 案例

```js
function outer() {
  let count = 0; // 外部局部变量
  return function inner() {
    // 闭包
    count++;
    console.log(count);
  };
}
const counter = outer();
counter(); // 1
counter(); // 2 (count变量被保留了)
```

✅ 应用场景

- `封装私有变量`
- `缓存计算结果`
- `函数柯里化（Currying）`
- `防抖与节流`

❌ 闭包的缺点

- **内存泄漏**：闭包会阻止垃圾回收，若使用不当会导致内存泄漏。
- **性能消耗**：闭包会对内存产生额外的消耗，可能会影响性能。

## this 指向

this 指向判断规则，按以下顺序依次判断，命中即停

| 步骤 | 判断条件                     | this 指向                                       |
| ---- | ---------------------------- | ----------------------------------------------- |
| 1    | 是箭头函数？                   | 定义时外层作用域的 `this`，不可修改             |
| 2    | 是 `new` 调用？                | 新创建的实例对象                                |
| 3    | 是 `call`/`apply`/`bind`？     | 指定的第一个参数（`null/undefined` 则退化为默认） |
| 4    | 是对象方法调用 `obj.fn()`？ | 最近的调用者对象                                |
| 5    | 以上都不是？                   | 默认绑定：严格模式 `undefined`，非严格 `window` |

## apply/call/bind

| 维度         | call                               | apply                                  | bind                                 |
| ------------ | ---------------------------------- | -------------------------------------- | ------------------------------------ |
| **执行时机** | 立即执行                           | 立即执行                               | 返回新函数，需要调用才执行           |
| **参数传递** | 逐个传递 <br/>`fn.call(ctx, a, b)` | 数组传递 <br/> `fn.apply(ctx, [a, b])` | 逐个传递<br/> `fn.bind(ctx, a, b)()` |
| **返回值**   | 函数执行结果                       | 函数执行结果                           | 绑定后的新函数                       |
| **柯里化**   | 不支持                             | 不支持                                 | 支持（预设参数）                     |

<Callout type="info" title="记忆口诀">
- **call**：逐个传参，立即执行（call → comma）
- **apply**：数组传参，立即执行（apply → array）
- **bind**：返回函数，稍后执行（bind → bind later）
</Callout>


## new 关键字

<Callout title="new 关键字做了哪些事"> 
`new` 操作符的执行过程：`创建空对象` → `链接原型` → `绑定 this，执行构造函数` → `返回对象`。
- 1.创建一个空对象
- 2.将空对象的 `__proto__` 指向函数的 `prototype`
- 3.将 `this` 绑定到空对象，然后执行构造函数
- 4.返回新对象
</Callout>

## Event Loop

<Callout type="info" title="核心规则">
① 执行同步代码 → ② 清空所有微任务 → ③ 下一个宏任务
</Callout>

<Callout type="warn" title="注意事项">
- ① 执行同步代码（其实就是第一个宏任务，它扮演的是第一个宏任务的角色）。
- 执行同步代码过程中，遇见宏任务放入宏任务队列中，遇见微任务放入微任务队列中。
- 真正的 Event Loop 过程是循环重复 ② ~ ③ 步骤。
- 同步代码扮演的是第一轮宏任务的角色，但它本身不走队列。
- 清空所有微任务过程中产生的微任务也在这轮清空
- await 后面的代码会被包装成微任务
</Callout>

### 执行模型

```
┌──────────────────────────┐
│       Call Stack          │
├──────────────────────────┤
│  同步代码执行             │
│  ↓ 清空后检查任务队列     │
├──────────────────────────┤
│  微任务（Microtask）      │ ← 优先、全部执行
│  · Promise.then/catch     │
│  · MutationObserver       │
│  · queueMicrotask         │
├──────────────────────────┤
│  宏任务（Macrotask）      │ ← 每次取一个
│  · setTimeout/setInterval │
│  · I/O / UI rendering     │
└──────────────────────────┘
```

### 执行顺序

1. 执行同步代码
2. 清空微任务队列
3. 取一个宏任务执行
4. 重复 2-3

### 经典题目

```js
console.log('start')
setTimeout(() => console.log('timeout'), 0)
Promise.resolve().then(() => console.log('promise'))
console.log('end')

// 输出: start → end → promise → timeout
```

## 事件委托（代理）

<Callout type="info" title="定义">
事件委托是`事件绑定在父元素上`上，利用`冒泡机制`统一处理子元素交互。
</Callout>

<Callout type="info" title="实现原理">
利用冒泡机制，把事件绑定在父元素上，通过 `event.target` 判断实际点击的元素。
</Callout>

<Callout type="info" title="DOM 事件传播">
DOM 事件传播分为三个阶段: `捕获阶段（从外到内）→ 目标阶段 → 冒泡阶段（从内到外）`。
- 捕获阶段：事件从 `window` 开始，沿着 DOM 树`向下`传播，直到**到达触发事件的目标元素**。
- 目标阶段：事件到达`真正触发事件的元素`
- 冒泡阶段：事件从`目标元素`开始，沿着 DOM 树`向上`传播到 `window`。
</Callout>


📙 事件冒泡与事件捕获过程

```sh
┌─────────────────────────────────────────┐
│                   document              │
│  ┌───────────────────────────────────┐  │
│  │                <html>             │  │
│  │  ┌─────────────────────────────┐  │  │
│  │  │            <body>           │  │  │
│  │  │  ┌───────────────────────┐  │  │  │
│  │  │  │        <div>          │  │  │  │
│  │  │  │  ┌─────────────────┐  │  │  │  │
│  │  │  │  │   <button>      │  │  │  │  │
│  │  │  │  │    Click Me     │  │  │  │  │
│  │  │  │  │   </button>     │  │  │  │  │
│  │  │  │  └─────────────────┘  │  │  │  │
│  │  │  └───────────────────────┘  │  │  │
│  │  └─────────────────────────────┘  │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘

点击 button 时，事件传播路径

1.捕获阶段（Capture Phase）
  document → html → body → div → button
                                  ↓
                              2.目标阶段（Target Phase）
                                  button（点击的元素本身）
                                  ↓
3.冒泡阶段（Bubble Phase）
  button → div → body → html → document
```

## 深拷贝和浅拷贝

<Callout type="info" title="浅拷贝与深拷贝的定义">
- 浅拷贝：只拷贝对象的第一层属性，若属性值是引用类型，则拷贝其`引用地址`，改一个引用会影响另一个。
- 深拷贝：递归拷贝对象的所有层级，创建完全独立的副本，互不影响。
</Callout>

📙 浅拷贝的实现方式

- 扩展运算符（`...`）
- `Object.assign()`
- 数组浅拷贝方法（`slice/concat/Array.from()`）
- 手动实现浅拷贝
- 第三方插件实现浅拷贝

📚 深拷贝的实现方式

- `JSON.parse(JSON.stringify())` — 常用
  - `JSON.stringify()` — 序列化（对象 -> 字符串）
  - `JSON.parse()` — 反序列化（字符串 -> 对象）
- `structuredClone()` — 现代方案
- `递归实现深拷贝` — 手写掌握
- `第三方库实现`

### 手写深拷贝要点

- 处理循环引用（WeakMap 记录已拷贝对象）
- 处理特殊对象（Date / RegExp / Map / Set）
- 处理函数（直接引用或跳过）

## == vs === vs Object.is()

- `==`: 松散相等，在比较两个值之前，若类型不统一（会`先强制转换型转换`）
- `===`: 严格相等，类型 和 值都相等（`不会进行类型转换`）
- `Object.is()`: 同值比较，比较两个值是否相等

## 0.1 + 0.2 !== 0.3

<Callout title="JS采用的双精度浮点数标准">
JavaScript 使用 `IEEE 754` 双精度浮点数标准，无法精确表示某些十进制小数。
</Callout>

✅ 解决方案

- 转为整数计算：先放大成整数，运算完再缩小，避开小数。
- 第三方库：`big.js`、`mathjs`、`decimal.js`、`bignumber.js`

## 脚本

### CSS与JS完整的阻塞关系图

```js
浏览器解析 HTML
    ↓
遇到 <link rel="stylesheet">
    ├── 继续解析 HTML（不阻塞）,并下载 CSS（异步）
    └── 构建 CSSOM 树（需要等待）
    ↓
遇到 <script>
    ├── 如果前面有未完成的 CSS
    │   └── 等待 CSS 加载完成（CSS 阻塞 JS）
    ├── 下载 JS
    ├── 执行 JS（阻塞 HTML 解析）
    └── 恢复解析
    ↓
等待 CSSOM 树构建完成
    ↓
合并 DOM + CSSOM = 渲染树
    ↓
渲染页面
```

### CSS加载会阻塞DOM渲染?

<Callout type="warn" title="答案">
会阻塞 DOM 渲染（CSS 不会阻塞 HTML 解析，但会阻塞 DOM 渲染）
</Callout>
```js
// 浏览器的处理流程
1. 解析 HTML，构建 DOM 树（CSS 不阻塞这个）
2. 下载并解析 CSS，构建 CSSOM 树（CSS 阻塞这个）
3. 等待 CSSOM 构建完成
4. 合并 DOM 树 + CSSOM 树，生成渲染树
5. 布局和绘制
6. 呈现页面

// 关键：渲染必须等待 CSSOM 构建完成
```

### JS加载会阻塞DOM渲染?

<Callout type="warn" title="答案">
JS 加载会阻塞 DOM 渲染
</Callout>

```js
// 浏览器解析流程
1. 解析 HTML，构建 DOM 树
2. 遇到 <script> 标签
   ├── 停止 HTML 解析（阻塞）
   ├── 下载 JS 文件（如果外部脚本）
   ├── 执行 JS 代码
   └── 恢复 HTML 解析
3. 继续解析后续 HTML
```

### 四种类型脚本对比

| 特性           | 普通 `script`  | `script async`                 | `script defer`   | `script type="module"`     |
| -------------- | -------------- | ------------------------------ | ---------------- | -------------------------- |
| 阻塞HTML解析   | ✅             | ❌ 否[下载时]，✅ 是[执行时]   | ❌ 否            | ❌ 否                      |
| 下载方式       | 同步下载       | 异步下载                       | 异步下载         | 异步下载                   |
| 下载时机       | 遇到立即下载   | 遇到并行下载                   | 遇到并行下载     | 遇到并行下载，同时分析依赖 |
| 执行时机       | 下载后立即执行 | 下载后立即执行                 | HTML解析完后执行 | HTML解析完后执行           |
| 执行顺序       | 文档顺序       | 下载完成顺序[谁先下完谁先执行] | 保证按文档顺序   | 文档顺序                   |
| 允许顶层 await | ❌             | ❌                             | ❌               | ✅ 需`async`标记           |

### 实际执行时机图示

```txt
HTML 解析 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━▶

普通脚本   [下载▼][执行▼]
           └阻塞─┘

async      [下载▼]  [执行▼]  (下载完就执行，可能打断解析)
           ←不阻塞→   └阻塞─┘

defer      [下载▼]              [执行▼]
           ←不阻塞→            DOM 解析完

module     [下载▼]              [执行▼]
(defer默认)←不阻塞→            DOM 解析完
```

### 详解

1️⃣ 普通 `script`

```js
<script src="app.js"></script>
```

- 浏览器解析 HTML 时，遇到它就会停下来。
- 如果是外链脚本，要先下载，再立即执行。
- 执行完后，才继续往下解析 HTML。
- 多个普通脚本严格按标签顺序执行。

<Callout type="warn" title="普通脚本的选择策略">
- 适合：必须立刻执行、并且依赖当前阻塞点上下文的脚本
- 缺点
 - 容易阻塞首屏渲染
 - 若脚本很大，页面会明显白屏/卡住
</Callout>

2️⃣ `script async`

```js
<script async src="analytics.js"></script>
```

- 遇到标签就并行下载，不阻塞 HTML 解析。
- 哪个脚本先下载完，就先执行。
- 执行时会打断 HTML 解析。
- 多个 async 之间没有顺序保证。
- DOMContentLoaded 不会专门等它。

<Callout type="warn" title="script async 选择策略">
- 适合(尽快下载，下载完就执行，顺序别指望)
  - 彼此独立的第三方脚本
  - 统计、埋点、广告、监控这类“不依赖页面其他脚本顺序”的资源
- 不适合
 - 有前后依赖关系的脚本
 - 依赖 DOM 已完整生成的初始化逻辑
</Callout>

3️⃣ `script defer`

```js
<script defer src="a.js"></script>
<script defer src="b.js"></script>
```

- 遇到标签就并行下载，不阻塞 HTML 解析。
- 等整个 HTML 解析完成后，再开始执行。
- 多个 defer 会按标签顺序执行。
- 会在 DOMContentLoaded 之前执行完，所以 DOMContentLoaded 会等它。

<Callout type="warn" title="script defer 优势">
- 适合（先下载，等 HMTL 文档解析完，然后按顺序执行）
- 应用场景
  - 主业务脚本
  - 依赖 DOM 基本结构已经就绪的初始化代码
  - 多个文件之间有明确执行顺序的场景
</Callout>

4️⃣ `script type="module"`

```js
<script type="module" src="main.js"></script>
```

- 默认就带有 defer-like 行为，不阻塞 `HTML` 解析。
- 会等 `HTML` 解析结束后执行。
- 支持 `import / export`，浏览器会继续拉取它的依赖模块。
- 模块有自己的作用域，不会把变量挂到全局。
- 默认是严格模式。
- 同一个模块 URL 通常只会执行一次并缓存。
- `DOMContentLoaded` 会等模块及其依赖执行完成。

## JS原生实现滚动加载

<Callout title="核心原理"> 
监听滚动事件，当滚动到底部（或接近底部）时，触发加载更多数据的逻辑。
</Callout>

### 判断触底

```txt
scrollHeight（内容总高度 = 整个盒子 = scrollTop + clientHeight + 剩余未显示高度 ）

┌─────────────────────────┐
│      scrollTop          │  ← 已滚动的距离/高度
│      (已滚过的部分)     │
├─────────────────────────┤  ← 可视区域顶部
│      clientHeight       │  ← 可视区域高度
│      (看得到的部分)     │
├─────────────────────────┤  ← 可视区域底部
│     剩余未show内容      │
└─────────────────────────┘
```

<Callout title="触底条件"> 
`scrollTop + clientHeight >= scrollHeight` (当滚动到底部时，剩余未显示高度 = 0)
</Callout>

### A方式：scroll 事件 + 节流

```js
const container = document.querySelector(".list-container");
let page = 1;
let loading = false;
let noMore = false;

// 节流函数
function throttle(fn, delay) {
  let timer = null;
  return function (...args) {
    if (timer) return;
    timer = setTimeout(() => {
      fn.apply(this, args);
      timer = null;
    }, delay);
  };
}

// 加载数据
async function loadMore() {
  if (loading || noMore) return;
  loading = true;

  try {
    const res = await fetch(`/api/list?page=${page}`);
    const data = await res.json();

    if (data.list.length === 0) {
      noMore = true;
      return;
    }

    // 渲染数据
    data.list.forEach((item) => {
      const div = document.createElement("div");
      div.textContent = item.name;
      container.appendChild(div);
    });

    page++;
  } finally {
    loading = false;
  }
}

// 监听滚动（带节流）
window.addEventListener(
  "scroll",
  throttle(() => {
    const { scrollTop, scrollHeight, clientHeight } = document.documentElement;
    // 距离底部 100px 时提前加载
    if (scrollTop + clientHeight >= scrollHeight - 100) {
      loadMore();
    }
  }, 200),
);

// 首次加载
loadMore();
```

### B方式：IntersectionObserver

<Callout title="核心思路（推荐）"> 
在列表底部放一个不可见的「哨兵元素」，用 `IntersectionObserver` 监听它是否进入视口，进入即触发加载。
</Callout>

**实现步骤**
- 1.放置哨兵：在列表容器底部放一个空的哨兵元素（`#sentinel`）
- 2.创建观察者：创建 `IntersectionObserver` 观察哨兵元素
- 3.触发条件判断：哨兵进入视口时（`isIntersecting` 为 `true`），调用 `loadMore` 加载下一页
- 4.防重复 & 结束标记：用 `loading` 防重复请求，`noMore` 标记数据已加载完毕
- 5.停止观察：数据加载完毕后调用 `observer.unobserve()` 停止观察
- 6.提前触发（可选）：通过 `rootMargin: '100px'` 实现提前触发，优化体验

```html tab="html"
<!--在列表底部放一个哨兵元素-->
<div class="article-list">
  <!-- 列表 -->
  <div v-for="article in list" :key="article.id" class="card">
    <h3>{{ article.title }}</h3>
    <p>{{ article.content }}</p>
  </div>

  <!-- 哨兵元素 -->
  <div ref="sentinelRef" id="sentinel"></div>

  <!-- 加载状态 -->
  <div v-if="loading" class="loading">加载中...</div>

  <!-- 数据加载完毕 -->
  <div v-if="noMore && list.length > 0" class="no-more">全部加载完毕</div>

  <!-- 错误处理 -->
  <div v-if="error" class="error" @click="refresh">
    加载失败，点击重试 ({{ error.message }})
  </div>

  <!-- 空状态 -->
  <div v-if="!loading && list.length === 0 && !error" class="empty">
    暂无数据
  </div>
</div>
```
```js tab="JS逻辑"
// 创建观察者
const observer = new IntersectionObserver((entries) => {
    // 哨兵进入视口时触发加载
    if (entries[0].isIntersecting) {
      loadMore();
    }
  },
  {
    rootMargin: "100px", // 提前 100px 触发
  },
);

const sentinel = document.getElementById("sentinel");
let page = 1;
let loading = false;
let noMore = false;

async function loadMore() {
  if (loading || noMore) return;
  loading = true;

  try {
    const res = await fetch(`/api/list?page=${page}`);
    const data = await res.json();

    if (data.list.length === 0) {
      noMore = true;
      sentinel.textContent = "没有更多了";
      observer.unobserve(sentinel); // 停止观察
      return;
    }

    const container = document.querySelector(".list-container");
    data.list.forEach((item) => {
      const div = document.createElement("div");
      div.textContent = item.name;
      container.appendChild(div);
    });

    page++;
  } finally {
    loading = false;
  }
}

observer.observe(sentinel);
loadMore();
```

### 两种方式对比

| 维度     | scroll + 节流              | IntersectionObserver          |
| -------- | -------------------------- | ----------------------------- |
| 性能     | 频繁触发，即使节流也有开销 | 浏览器原生优化，性能更好      |
| 代码复杂 | 需要手动计算位置 + 节流    | 回调直接返回 `isIntersecting` |
| 兼容性   | 全兼容                     | 现代浏览器均支持              |
| 精度     | 取决于节流间隔             | 精确                          |
| 推荐     | 需要兼容老浏览器时         | ✅ 优先使用                   |
| 使用场景 | 复杂滚动逻辑               | 可见性检测专用                |

### 关键细节

- **防重复请求**：用 `loading` 标志位，请求中不再触发
- **没有更多数据**：用 `noMore` 标志位，到底后停止请求
- **提前加载**：不要等完全触底，提前 100-200px 开始加载，体验更好
- **节流而非防抖**：滚动场景用节流更合适，保证持续滚动时也能触发
- **关键配置**
  - `rootMargin` 提前触发
  - `isLoading` 防重复
  - 数据加载完毕后 `unobserve` 及时释放资源。
