## 组件与插件

| 维度     | 组件 (Component) | 插件 (Plugin)       |
| -------- | ---------------- | ------------------- |
| 作用     | 构建 UI 片段     | 扩展 Vue 全局功能   |
| 目标     | App.vue          | Vue 本身            |
| 注册     | 局部 / 全局      | `app.use(plugin)`   |
| 编写形式 | `.vue` 文件      | 暴露 `install` 方法 |

### 插件 install 核心对照

| 功能           | Vue 2 写法           | Vue 3 写法                         |
| -------------- | -------------------- | ---------------------------------- |
| 全局属性/方法  | `Vue.prototype.$xxx` | `app.config.globalProperties.$xxx` |
| 全局组件       | `Vue.component()`    | `app.component()`                  |
| 全局指令       | `Vue.directive()`    | `app.directive()`                  |
| provide/inject | 手动实现             | `app.provide()` 原生支持           |

<Callout type="warn">
  Vue 3 插件注册必须在 `app.mount()` 之前，否则不生效。
</Callout>

## 生命周期

### Vue 2

| 钩子            | 可访问 data | 可操作 DOM | 适用场景                       |
| --------------- | ----------- | ---------- | ------------------------------ |
| `beforeCreate`  | 否          | 否         | 初始化前（几乎不用）           |
| `created`       | 是          | 否         | 数据请求、初始化               |
| `beforeMount`   | 是          | 否         | 挂载前的最后数据修改           |
| `mounted`       | 是          | 是         | DOM 操作、第三方库初始化       |
| `beforeUpdate`  | 是          | 是         | 更新前获取旧 DOM 状态          |
| `updated`       | 是          | 是         | 操作更新后的 DOM               |
| `beforeDestroy` | 是          | 是         | 清理定时器、移除监听（最重要） |
| `destroyed`     | 是          | 否         | 清理完成的通知                 |
| `activated`     | 是          | 是         | keep-alive 激活                |
| `deactivated`   | 是          | 是         | keep-alive 失活                |
| `errorCaptured` | 是          | 是         | 捕获子组件错误                 |

### Vue 3

| Options API       | Composition API     | 适用场景                         |
| ----------------- | ------------------- | -------------------------------- |
| `beforeCreate`    | `setup()`           | 初始化前（几乎不用，setup 替代） |
| `created`         | `setup()`           | 数据请求、初始化（setup 替代）   |
| `beforeMount`     | `onBeforeMount`     | 挂载前的最后数据修改             |
| `mounted`         | `onMounted`         | DOM 操作、第三方库初始化         |
| `beforeUpdate`    | `onBeforeUpdate`    | 更新前获取旧 DOM 状态            |
| `updated`         | `onUpdated`         | 操作更新后的 DOM                 |
| `beforeUnmount`   | `onBeforeUnmount`   | 清理资源（最重要）               |
| `unmounted`       | `onUnmounted`       | 清理完成的通知                   |
| `activated`       | `onActivated`       | keep-alive 激活                  |
| `deactivated`     | `onDeactivated`     | keep-alive 失活                  |
| `errorCaptured`   | `onErrorCaptured`   | 捕获子组件错误                   |
| `renderTracked`   | `onRenderTracked`   | 调试用，追踪响应式依赖（dev）    |
| `renderTriggered` | `onRenderTriggered` | 调试用，追踪重渲染触发（dev）    |

<Callout type="info" title="keep-alive 缓存组件">
  - 首次进入：`created → mounted → activated` - 再次进入：`activated`（不经过
  created/mounted） - 失活：`deactivated`（不经过 destroy）
</Callout>

## data 为什么是函数

- **根实例**：data 可以是对象（单例，不复用）
- **组件**：data 必须是函数（防止多个实例共享同一引用）

<Callout>
  组件复用时，如果 data 是对象，所有实例引用同一份数据 →
  数据污染。函数每次返回新对象，保证独立性。
</Callout>

## 虚拟 DOM

- `真实 DOM`：浏览器中渲染的实际节点对象，直接操作开销大。
- `虚拟 DOM`：虚拟 DOM 是真实 DOM 的 JS 对象映射。

📙 `虚拟 DOM`的核心价值

<Callout>
  当数据变更时，通过新旧两棵虚拟 DOM 树的 Diff
  算法，计算出最小差异，再批量应用到真实 DOM 上，以此减少昂贵的 DOM 操作。
</Callout>

## Diff 算法

<Callout type="info" title="概念">
  Diff 算法是同层级比较新旧 VNode 子节点的算法，找出最小差异，再批量更新到真实
  DOM。 - Vue2 用双端比较（头头、尾尾、头尾、尾头） - Vue3 用快速 Diff (Fast
  Diff) + 最长递增子序列。
</Callout>

## v-for 中 key 的作用

<Callout type="info" title="key 的作用">
  key 是节点的唯一标识，帮助 Diff
  算法`精准识别节点复用`，避免复用带来的渲染错误问题，同时提升更新效率。
</Callout>

## watch vs watchEffect

1️⃣ 核心区别（极简版）

| 维度       | watch          | watchEffect               |
| ---------- | -------------- | ------------------------- |
| 依赖收集   | 显式指定监听源 | 自动收集                  |
| 默认执行   | 懒执行         | 立即执行                  |
| 拿到旧值   | ✅             | ❌                        |
| 控制精度   | 高             | 中                        |
| 异步稳定性 | 更稳           | 有陷阱（仅跟踪 await 前） |

2️⃣ `watchEffect` 高频追问速记

- 为何立即执行？ → 为了`首次执行 + 依赖收集`。
- 为什么触发过多？ → 回调中读到的`所有响应式数据`都会成为依赖。
- 异步陷阱？ → 只跟踪 `await 之前`的依赖。
- 连续请求污染咋办？ → 用 `onCleanup` 取消上一次的副作用。

3️⃣ 使用场景

- `watch`：需要旧值、明确指定监听源、复杂异步、延迟执行。
- `watchEffect`：依赖多、自动跟踪、执行时响应式数据用到谁就监听谁、初始化执行。

<Callout type="info" title="停止监听">
`watch` 和 `watchEffect` 都返回一个停止函数，调用返回函数即可停止监听；
在 `setup`/ `<script setup>` 中使用时，组件卸载会自动停止，通常无需手动处理。
</Callout>

## computed vs watch

1️⃣ 核心区别（极简版）

| 维度   | computed                                   | watch          |
| ------ | ------------------------------------------ | -------------- |
| 定位   | 派生新值                                   | 执行副作用     |
| 返回值 | 有（计算结果）                             | 无（停止函数） |
| 缓存   | 有（依赖项变化重新计算，否是返回缓存结果） | 无             |
| 异步   | ❌ 不适合                                  | ✅ 适合        |
| 副作用 | ❌ 不应有                                  | ✅ 专为其设计  |

2️⃣ 使用场景

- `computed`：模板加工、组合状态、过滤排序、状态派生、结果计算、表单校验。
- `watch`：请求、缓存、路由联动、埋点、第三方实例操作。

3️⃣ `computed` 和 `watch` 都能监听数据变化，怎么选择？

- `computed`：用于`派生`数据，有返回值，可缓存计算结果。示例：fullName/totalPrice
- `watch`：用于`副作用`，无返回值。示例：发请求/操作 DOM/路由跳转

## computed vs methods

- `computed` 基于响应式依赖派生出新状态，只有依赖变化时才重新计算，否则使用缓存。
- `methods` 每次调用都会执行，无缓存机制。

📚 computed vs methods（核心对比）

| 对比维度         | computed                     | methods                    |
| :--------------- | :--------------------------- | :------------------------- |
| **缓存机制**     | ✅ 有缓存，依赖不变不重算    | ❌ 无缓存，每次调用都执行  |
| **触发时机**     | 依赖的响应式数据变化时       | 调用时立即执行             |
| **性能**         | 高（缓存减少无用计算）       | 低（频繁调用可能浪费性能） |
| **是否支持传参** | ❌ 不支持                    | ✅ 支持                    |
| **返回值**       | 必须返回一个值               | 可有可无                   |
| **使用场景**     | 过滤排序、状态派生、结果计算 | 事件处理/复杂逻辑/传参计算 |

## watch vs methods

- `watch`：数据变化时自动触发
- `methods`：每次调用都会触发

📚 watch vs methods（核心对比）

| 对比维度             | watch                       | methods                    |
| :------------------- | :-------------------------- | :------------------------- |
| **触发方式**         | 数据变化时自动触发          | 主动触发                   |
| **执行时机**         | 数据变化后立即执行          | 调用时立即执行             |
| **是否支持异步**     | ✅ 支持                     | ✅ 支持                    |
| **是否支持深度监听** | ✅ 支持 `deep: true`        | ❌ 无此概念                |
| **初始执行**         | ✅ 支持 `immediate: true`   | ❌ 无此概念                |
| **使用场景**         | 数据变化后的副作用/异步请求 | 事件处理/复用逻辑/普通函数 |

## computed/watch/methods

| 对比维度               | computed                  | watch               | methods           |
| :--------------------- | :------------------------ | :------------------ | :---------------- |
| **是支持缓存**         | ✅ 是                     | ❌ 否               | ❌ 否             |
| **触发方式**           | 自动（依赖变化）          | 自动（数据变化）    | 主动（调用）      |
| **执行时机**           | 读取时/依赖项目变化是执行 | 数据变化时执行      | 调用时执行        |
| **是否支持传参**       | ❌ 否                     | ❌ 否               | ✅ 是             |
| **是否必须返回值**     | ✅ 是                     | ❌ 否               | ❌ 否             |
| **是否适合异步**       | ❌ 否                     | ✅ 是               | ✅ 是             |
| **是否支持 deep**      | ✅ 是（访问深层时自动）   | ✅ 是（手动配置）   | ❌ 否             |
| **是否支持 immediate** | ❌ 否                     | ✅ 是               | ❌ 否             |
| **典型场景**           | 模板计算/派生状态         | 数据变化副作用/请求 | 事件处理/复用逻辑 |

## 响应式原理

### Object.defineProperty

- 初始化时遍历 `data` 的每个属性，添加 `getter/setter`
- **无法监听新增属性** → 需要 `Vue.set()`
- **无法监听 delete** → 需要 `Vue.delete()`
- **无法监听数组索引赋值** → 性能考虑，未对数组索引做劫持
- **无法监听数组 length 变化** → `length` 不是响应式属性
- vue2 重写了 `push`/`pop`/`shift`/`unshift`/`splice`/`sort`/`reverse` 七个方法

### Proxy

- 代理整个对象，拦截所有操作
- 自动监听新增属性和删除操作
- 惰性代理：访问到哪层才代理哪层
- 原生支持数组索引和 `length` 变化

### Vue 3 响应式流程

```text
数据读取 (get)
  ↓
track(target, key) — 收集 effect 到依赖 Map
  ↓
数据修改 (set)
  ↓
trigger(target, key) — 找到并执行依赖中的 effect
  ↓
组件重新渲染 / computed 重新计算 / watch 回调
```

## ref vs reactive

| 特性       | `ref`       | `reactive`         |
| ---------- | ----------- | ------------------ |
| 适用类型   | 基础 + 引用 | 仅对象             |
| 访问方式   | `.value`    | 直接访问属性       |
| 替换整体值 | 支持        | 不支持（丢失代理） |
| 解构       | 安全        | 丢失响应式         |

<Callout>
  Vue 3.2+ 官方推荐**优先使用 ref**：统一 `.value`
  语法，无解构丢失问题，适合整体替换。
</Callout>

## v-show vs v-if

| 维度     | `v-if`         | `v-show`        |
| -------- | -------------- | --------------- |
| 控制     | 添加/删除 DOM  | `display: none` |
| 初始渲染 | 条件为真才渲染 | 总是渲染        |
| 切换开销 | 高（重建 DOM） | 低（切换样式）  |
| 适用场景 | 条件很少变化   | 频繁切换        |
| 生命周期 | 触发           | 不触发          |

## v-if 与 v-for 优先级

| 版本  | 优先级         | 说明         |
| ----- | -------------- | ------------ |
| Vue 2 | `v-for > v-if` | 先遍历再判断 |
| Vue 3 | `v-if > v-for` | 先判断再遍历 |

<Callout type="warn">
  不要在同一元素上使用 `v-if` 和 `v-for`。用 `computed` 过滤列表或将 `v-if`
  移到外层。
</Callout>

## nextTick

### 概念

`nextTick` 是 Vue 提供的 `异步 DOM 更新机制`，**用于在下次 DOM 更新循环结束后执行回调**，确保能获取到更新后的 DOM。

```js
msg.value = "Vue3 NextTick!";
console.log(box.value?.textContent); // 可能是旧值

nextTick(() => {
  console.log(box.value?.textContent); // 确保拿到新值
});
```

### 使用场景

- 修改数据后读取更新后的 DOM
- `v-if` 创建元素后初始化第三方库
- 动态显示输入框后设置焦点

## Vue Router

### 导航守卫

| 类型         | 守卫                                                          |
| ------------ | ------------------------------------------------------------- |
| 全局前置守卫 | `beforeEach`                                                  |
| 全局解析守卫 | `beforeResolve`                                               |
| 全局后置守卫 | `afterEach`                                                   |
| 路由独享     | `beforeEnter`                                                 |

### 路由传参

- `path`+ `query` 传参: 在浏览器地址栏中显示参数，类似`get`传参
- `name` + `params` 传参: 在浏览器地址栏中不显示参数，类似`post`传参

<Callout type="warn" title="注意事项">
  - 刷新：不会丢失 `query` 里面的数据 - 刷新：会丢失 `params` 里面的数据
</Callout>

### $route 和 $router

| 维度         | `$route`                              | `$router`                        |
| ------------ | ------------------------------------- | -------------------------------- |
| **本质**     | 当前激活的路由**信息对象**            | 整个路由**实例**                 |
| **属性**     | `path`/`params`/`query`/`name`/`meta` | `push`/`replace`/`go`/`addRoute` |
| **职责**     | 只读，获取当前路由状态                | 读写，控制路由跳转               |
| **是否可变** | 每次导航产生新对象                    | 全局唯一实例                     |

### 动态路由+路由权限

📚 常见的三种实现方案

| 方案                          | 核心思路                                                          | 适用场景             |
| ----------------------------- | ----------------------------------------------------------------- | -------------------- |
| **① 后端返回完整路由**        | 后端存路由配置，前端 `import.meta.glob` 映射组件，`addRoute` 注册 | 权限频繁变动的中后台 |
| **② 前端全量路由 + 权限过滤** | 前端定义全部路由 + `meta.roles`，后端返回角色标识过滤             | 角色固定/页面少      |
| **③ 公共路由 + 后端菜单**     | 公共路由内置在前端，业务路由后端返回动态追加                      | 多角色/公共页面统一  |

1️⃣ 方案一：后端返回完整路由

```
登录 → 获取 Token → 请求后端菜单接口 → 后端返回树形路由结构 → 通过 addRoute 动态注册路由
```

- **原理**：后端存储路由配置（path、component、meta），前端拿字符串路径通过 `import.meta.glob` 映射为组件
- **优点**：权限完全由后端控制，前端改动少，角色/菜单变更无需发版
- **缺点**：前后端需约定 component 命名规范，新增页面需后端同步配置
- **适用**：菜单结构复杂、权限变动频繁的中后台系统

2️⃣ 方案二：前端全量路由 + 后端返回权限标识过滤

```
登录 → 获取角色/权限标识 → 前端全量路由按 meta.roles 过滤 → 动态注册 → 渲染菜单
```

- **原理**：前端定义全部路由，每条路由标注 `meta.roles`，后端只返回用户拥有的权限/角色编码，前端按编码过滤
- **优点**：路由配置集中在前端，后端只需管角色分配；路由跳转逻辑更直观
- **缺点**：新增页面需前端编码+发版；权限粒度为角色级，颗粒度粗
- **适用**：角色固定、页面较少的管理系统

3️⃣ 方案三：公共路由 + 后端返回菜单路由

```
登录 → 初始化前端公共路由（内置） → 请求菜单接口 → 后端返回动态路由部分 → 通过 addRoute 动态注册路由
```

- **原理**：`createRouter` 时注册公共路由（静态），守卫中请求后端获取用户专属路由，追加注册
- **优点**：公共页面不受权限影响（如首页、404），仅业务路由动态
- **缺点**：与方案一类似，需后端同步路由配置
- **适用**：多角色但公共页面统一的系统

📙 方案对比

| 维度         | 后端路由               | 前端过滤                     | 混合模式            |
| ------------ | ---------------------- | ---------------------------- | ------------------- |
| 路由配置存放 | 后端数据库             | 前端代码                     | 公共前端 + 业务后端 |
| 菜单变更成本 | 改数据库即生效         | 前端改代码+发版              | 公共发版，业务改库  |
| 权限粒度     | 菜单+按钮级            | 路由+角色级                  | 菜单+按钮级         |
| 安全性       | 高（无权限路由不注册） | 中（路由仍注册，靠守卫拦截） | 高                  |
| 实施复杂度   | 中（需约定组件映射）   | 低                           | 中                  |

### hash vs history 模式

## Vuex vs Pinia

| 特性     | Vuex             | Pinia                   |
| -------- | ---------------- | ----------------------- |
| Mutation | 必须有，同步修改 | 已移除，Action 统一处理 |
| 模块化   | 嵌套 modules     | 平铺独立 store          |
| TS 支持  | 较弱             | 原生支持                |
| 动态注册 | 需手动           | 默认动态                |

### Vuex 核心

- **State**：状态数据
- **Getter**：派生状态
- **Mutation**：修改 `State` 状态的唯一方法
- **Action**：同步 + 异步操作（提交 `Mutation`）
- **modules**：模块化

### Pinia 核心

- **State**：状态数据
- **Getter**：派生状态
- **Action**：同步 + 异步操作（不再区分 mutation）

## 性能优化

### SPA 首屏优化

| 优化方向     | 方案                       |
| ------------ | -------------------------- |
| 减小入口体积 | 路由懒加载                 |
| 静态资源缓存 | HTTP 缓存 + Service Worker |
| UI 框架按需  | Tree Shaking / 自动导入    |
| 图片优化     | WebP / 懒加载 / 响应式图片 |
| GZip 压缩    | Nginx / CDN 开启           |
| SSR          | 服务端渲染首屏             |

## Vue 设计模式

| 模式       | Vue 应用                    | 重要程度 |
| ---------- | --------------------------- | -------- |
| 观察者模式 | 响应式系统、watch、computed | 核心     |
| 代理模式   | Proxy 响应式、ref           | 核心     |
| 组合模式   | 组件树、插槽                | 核心     |
| 工厂模式   | `createApp`、`h` 函数       | 重要     |
| 策略模式   | `v-if/v-show`、过渡动画     | 重要     |
| 发布订阅   | 事件总线、mitt              | 重要     |
| 单例模式   | Router、Store               | 一般     |
