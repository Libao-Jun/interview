## MVVM 架构

### 概述
MVVM（Model-View-ViewModel：模型-视图-视图模型）：是一种软件架构模式，旨在将应用程序的业务逻辑与用户界面（UI）清晰地分开。

:::tip[Vue 是 MVVM 吗？]
答案：不是，仅**受 MVVM 启发**，但不完全遵循，核心是**响应式系统**
:::
### Vue 中的 MVVM 体现
| 层级      | 职责                   |  表现 | 说明                           |
| --------- | -----------------------|-- | ------------------------------ |
| Model     | 数据存储、业务逻辑 | 数据状态  | API 服务 / Pinia / Vuex |
| View      | UI展示和用户交互       |  HTML 结构、CSS 样式、DOM 元素     | `<template>` 模板     |
| ViewModel | 数据与视图的桥梁|响应式数据、计算属性、侦听器、事件、生命周期钩子 | Vue 实例 / `setup()` 返回值    |

:::info[Model (模型)]
- data 初始数据
- Pinia/Vuex
- 本地存储
- 后端API
:::
:::info[View (视图) ]
- template 模板（DOM 元素、CSS 样式）
:::
:::info[ViewModel (视图模型)]
- 响应式数据（data / ref / reactive）
- 计算属性（computed）
- 事件（methods）
- 侦听器（watch）
- 生命周期钩子
:::

## SPA vs MPA

1️⃣ 核心定义

:::info[单页应用（SPA，Single Page Application）]
整个应用只有一个 HTML 页面，内容的更新和切换通过 **前端路由** 来实现，而不会重新加载整个页面。
:::

:::info[多页面应用（MPA，Multi-Page Application）]
每次页面跳转时，后台服务器都会返回一个新的 HTML 页面，页面加载是 **整页刷新** 的。
:::

2️⃣ 详细对比表

| 维度       | 单页应用（SPA）                                                      | 多页面应用（MPA）                                      |
| ---------- | -------------------------------------------------------------------- | ------------------------------------------------------ |
| 核心定义   | 仅加载一个 HTML 页面，内容切换靠前端路由和 JS 动态渲染，无需整页刷新 | 每次跳转都从服务器请求新 HTML 页面，页面切换会整页刷新 |
| 首屏加载   | 首次加载资源较多（JS/CSS），后续切换快                               | 每次页面跳转都需重新加载资源，首屏快，后续慢           |
| 路由实现   | 前端路由（如 Vue Router），URL 变化不刷新页面                        | 传统后端路由，URL 变化会刷新整个页面                   |
| 用户体验   | 切换流畅、无闪烁、                                                   | 切换有白屏或闪烁，体验传统                             |
| SEO 友好性 | 需配合 SSR/预渲染优化，默认对 SEO 不友好                             | 天生支持 SEO，爬虫可直接抓取每个页面                   |
| 资源管理   | 资源一次性加载，适合组件化、模块化开发                               | 每页独立资源，适合大型门户、内容型网站                 |
| 状态管理   | 需前端集中管理（如 Vuex/Pinia），页面间状态可共享                    | 每页独立刷新，状态天然隔离，依赖后端 session/cookie    |
| 开发复杂度 | 前端工程化要求高，需处理路由、权限、缓存、首屏优化等                 | 结构简单，开发门槛低，适合传统后端开发                 |
| 适用场景   | 后台管理、WebApp、交互性强的应用                                     | 门户网站、新闻资讯、SEO 要求高的内容型网站             |
| 缺点       | 首屏慢、SEO 难、首包大、浏览器兼容性需关注                           | 页面跳转慢、体验差、代码复用难、前后端协作复杂         |

3️⃣ 主要异同点总结

- 相同点：都属于 Web 应用，最终都在浏览器中运行。
- 不同点：
  - SPA 只加载一次页面，依赖 JS 动态渲染，体验好但 SEO 差；
  - MPA 每次跳转都刷新页面，体验传统但 SEO 友好。

## 组件与插件

### 组件 vs 插件

| 维度     | 组件 (Component) | 插件 (Plugin)       |
| -------- | ---------------- | ------------------- |
| 作用     | 构建 UI 片段     | 扩展 Vue 全局功能   |
| 目标     | App.vue          | Vue 本身            |
| 注册     | 局部 / 全局      | `app.use(plugin)`   |
| 编写形式 | `.vue` 文件      | 暴露 `install` 方法 |

### Vue2 vs Vue3 install

| 功能           | Vue 2 写法           | Vue 3 写法                         |
| -------------- | -------------------- | ---------------------------------- |
| 全局属性/方法  | `Vue.prototype.$xxx` | `app.config.globalProperties.$xxx` |
| 全局组件       | `Vue.component()`    | `app.component()`                  |
| 全局指令       | `Vue.directive()`    | `app.directive()`                  |
| provide/inject | 手动实现             | `app.provide()` 原生支持           |

:::warning
Vue 3 插件注册必须在 `app.mount()` 之前，否则不生效。
:::

## 组件的命名规范
- 组件名应该始终是多个单词。
- 组件的 `data` 必须是一个函数。
- 为组件样式设置作用域。
- 和父组件紧密耦合的子组件应该以父组件名作为前缀命名。
- 组件名应该以高级别的 (通常是一般化描述的) 单词开头，以描述性的修饰词结尾。
- 模板中的组件名大小写（推荐：kebab-case）。
- 组件名应该倾向于完整单词而不是缩写。

:::info[单文件组件命名规范（强烈推荐）]
- 大驼峰命名（PascalCase）
- 横线连接 (kebab-case)
:::

## 生命周期

### Vue 2

| 钩子            | 可访问 data | 可操作 DOM | 适用场景                       |
| --------------- | ----------- | ---------- | ------------------------------ |
| `beforeCreate`  | 否          | 否         | 初始化前     |
| `created`       | 是          | 否         | 异步请求、初始化               |
| `beforeMount`   | 是          | 否         | 挂载前的最后数据修改           |
| `mounted`       | 是          | 是         | DOM 操作、第三方库初始化       |
| `beforeUpdate`  | 是          | 是         | 更新前获取旧 DOM 状态          |
| `updated`       | 是          | 是         | 操作更新后的 DOM               |
| `beforeDestroy` | 是          | 是         | 清理定时器、移除监听（最重要） |
| `destroyed`     | 是          | 否         | 清理完成的通知                 |
| `activated`     | 是          | 是         | keep-alive 激活                |
| `deactivated`   | 是          | 是         | keep-alive 失活                |
| `errorCaptured` | 是          | 是         | 捕获子组件错误                 |

:::info[常用场景]
- beforeCreate（创建之前）：几乎不使用，只有极少数插件初始化场景
- `data`、`methods`、`DOM`不可用
- created（创建完成）：① 异步请求数据（API 调用） ② 初始化非响应式数据 ③ 设置定时器
- 此时不能操作 `DOM`，如需操作需使用 `$nextTick`
- beforeMount（挂载之前）：① 最后一次修改数据的机会 ② 服务端渲染（SSR）中使用
- 很少使用，因为数据修改不会触发更新，且 `$el` 不可用
- mounted（挂载完成）：① DOM 操作 ② 第三方库初始化（如 ECharts、地图） ③ 添加监听事件
- 子组件已全部挂载完成，可以安全操作 `DOM`
- beforeUpdate（更新之前）：① 在 `DOM` 更新前做最后的修改 ② 移除旧事件监听
- 不要在此钩子中修改数据，否则会触发无限循环
- updated（更新完成）：① 依赖 `DOM` 更新的操作 ② 更新第三方库实例
- 不要在此钩子中修改数据，会触发无限循环；如需修改需用条件判断
- beforeDestroy（销毁之前）：① 清除定时器 ② 移除事件监听 ③ 销毁第三方库实例
- 必须清理副作用，否则会造成内存泄漏
- destroyed（销毁完成）：① 清理最后的资源 ② 日志上报
- 此时组件已完全销毁，不应再操作` DOM` 或访问响应式数据
:::

### Vue 3

| Options API       | Composition API     | 适用场景                         |
| ----------------- | ------------------- | -------------------------------- |
| `beforeCreate`    | `setup()`           | 初始化前（几乎不用，setup 替代） |
| `created`         | `setup()`           | 数据请求、初始化（setup 替代）   |
| `beforeMount`     | `onBeforeMount`     | 挂载前的最后数据修改             |
| `mounted`         | `onMounted`         | DOM 操作、第三方库初始化         |
| `beforeUpdate`    | `onBeforeUpdate`    | 更新前获取旧 DOM 状态            |
| `updated`         | `onUpdated`         | 操作更新后的 DOM                 |
| `beforeUnmount`   | `onBeforeUnmount`   | 清理资源（最重要）               |
| `unmounted`       | `onUnmounted`       | 清理完成的通知                   |
| `activated`       | `onActivated`       | keep-alive 激活                  |
| `deactivated`     | `onDeactivated`     | keep-alive 失活                  |
| `errorCaptured`   | `onErrorCaptured`   | 捕获子组件错误                   |
| `renderTracked`   | `onRenderTracked`   | 调试用，追踪响应式依赖（dev）    |
| `renderTriggered` | `onRenderTriggered` | 调试用，追踪重渲染触发（dev）    |

:::info[keep-alive 缓存组件]
- 首次进入：`beforeCreate → created → mounted → activated` 
- 再次进入：`activated`（不经过 `beforeCreate / created / mounted`） 
- 失活：`deactivated`（不经过 `destroy`）
:::

## 虚拟 DOM

### 概念

- `真实 DOM`：浏览器中渲染的实际节点对象，直接操作开销大。
- `虚拟 DOM`：虚拟 DOM 是真实 DOM 的 JS 对象映射。

📙 `虚拟 DOM`的核心价值

:::info
当数据变更时，通过 **Diff 算法** 比较新旧两棵虚拟 DOM 树，计算出最小差异，
再**批量**应用到真实 DOM 上，以此减少昂贵的 DOM 操作。
:::
### 核心流程

```txt
数据变化 → 重新执行 render → 新 VNode
         ↓
       diff（新旧 VNode 对比）
         ↓
       patch（计算出最小化差异，更新真实 DOM）
```

### 渲染流程

```sh
模板/渲染函数
  ↓
编译器 compile()
  ↓
渲染函数 render()
  ↓
创建 VNode（虚拟 DOM 节点）
  ↓
渲染器 renderer
  ↓  首次渲染
创建真实 DOM（mount）
  ↓  后续更新
Diff 新旧 VNode → 生成补丁（patch）
  ↓
最小化更新真实 DOM
```

## Diff 算法

:::info[概念]
Diff 算法是同层级比较新旧 VNode 子节点的算法，找出最小差异，再批量更新到真实 DOM。
- Vue2 用双端比较（头头、尾尾、头尾、尾头） 
- Vue3 用快速 Diff (Fast Diff) + 最长递增子序列。
:::

## 编译优化

:::info
Vue 3 编译器在编译期分析模板，生成带优化标记的渲染函数，减少运行时开销。
:::

| 优化手段     | 原理                          | 收益               |
| ------------ | ----------------------------- | ------------------ |
| 静态提升     | 不变节点只创建一次            | 减少创建/对比开销  |
| 补丁标记     | 位标记仅 `diff` 动态部分      | 跳过静态节点       |
| 块级树       | 扁平化动态子节点列表          | O(全部) → O(动态)  |
| 事件缓存     | 缓存事件处理函数引用          | 避免子组件无效更新 |
| 静态属性提升 | 不变的 `props` 对象只创建一次 | 减少对象创建       |

## 组合式函数（Composables）

### 概念

:::info[组合式函数]
组合式函数(Composables) 是一个利用 Vue 的`组合式 API`
来封装和复用`有状态逻辑`的函数。
:::

### 约定和最佳实践

- 命名​：约定用`驼峰命名法`命名，并以 `use` 作为开头。
- 返回值（推荐）：**始终返回一个包含多个 ref 的普通的非响应式对象**。
- 使用限制：只能在 `<script setup>` 或 `setup()` 钩子中被调用（它们也只能被`同步`调用）

### Composables vs mixins

| 维度             | Mixin（混入）                                                                  | Composables（组合式函数）              |
| ---------------- | ------------------------------------------------------------------------------ | -------------------------------------- |
| **数据来源追溯** | 多个 mixin 的属性合并到组件实例，来源不清，<br/>追溯实现和理解组件行为变得困难 | 显式解构赋值，属性来自哪个函数一目了然 |
| **命名冲突**     | 多 mixin 可能注册同名属性，造成冲突，排查困难                                  | 解构时可重命名，编译阶段即可避免       |
| **隐式耦合**     | 多个 mixin 通过共享属性名通信，使它们隐性地耦合在一起                          | 函数传参调用，依赖关系显式声明         |
| **类型支持**     | 弱，实例上的属性难以推导                                                       | 强，TS 类型自动推导                    |
| **Tree Shaking** | 不支持                                                                         | 支持，按需引入                         |
| **代码组织**     | 按选项分散（data/methods/watch）                                               | 按功能聚合在一起                       |

:::info[Vue2 对 Mixin中同名的实际处理机制]
📙 同名处理机制

- 同名 `生命周期钩子`会合并为数组，依次执行（mixin 的先执行，组件自身的后执行）
- 同名 `data/props/methods/computed` 等，`组件自身的优先级高于 mixin（即组件自身的覆盖 mixin 的，只保留一个）`
- 多个 mixin 之间的同名属性，`后注册的覆盖先注册的`

📚 表格总结

| 类型                 | 处理方式                                   | 执行顺序 / 优先级    |
| -------------------- | ------------------------------------------ | -------------------- |
| **生命周期钩子**     | 合并为数组，依次调用（非生命周期触发多次） | mixin 先 → 组件后    |
| **data/props**       | 组件覆盖 mixin                             | 组件优先，只保留一个 |
| **methods/computed** | 组件覆盖 mixin                             | 组件优先，只保留一个 |
| **watch**            | 合并为数组，依次调用（非 watch 触发多次）  | mixin 先 → 组件后    |

⚠ 注意事项（合并机制）

- 生命周期钩子：生命周期钩子中的`函数体`合并为数组，生命周期`只触发一次`，仅会依次调用合并后数组里的`函数体`。
- watch：`watch`中的`副作用内容`合并为数组，`watch只触发一次`，仅依次调用合并后数组里的`副作用内容`。

:::

## 代码组织顺序

```ts
imports（导入）
  ↓
defineOptions（组件选项：name、inheritAttrs 等）
  ↓
interface / type（局部类型声明）
  ↓
defineProps / defineEmits / defineSlots（组件契约）
  ↓
composables（useXxx 组合式函数）
  ↓
state（ref / reactive 变量定义）
  ↓
computed（计算属性）
  ↓
functions（事件处理、业务方法）
  ↓
watch / watchEffect（侦听器）
  ↓
lifecycle（onMounted、onUnmounted 等）
  ↓
defineExpose（暴露给父组件的 API，放最后，一眼看出对外接口）
```

## data 为什么是函数

- **根实例**：data 可以是对象（单例，不复用）
- **组件**：data 必须是函数（防止多个实例共享同一引用）

:::info
组件复用时，如果 data 是对象，所有实例引用同一份数据 →
数据污染。函数每次返回新对象，保证独立性。
:::

## methods 不能用箭头函数

📚 this 指向问题（根本原因）

- 是因为箭头函数的 `this` 指向定义时的外层作用域的 `this`，
- 导致`无法正确指向 Vue 实例`，进而不能访问 `data`、`methods` 等组件属性。

:::info[Vue 内部绑定机制]
- Vue 在实例化时，会遍历 `methods` 对象，用 `bind` 将每个方法的 `this` 绑定到
Vue 实例。 - 但箭头函数无法被 `bind` 改变 `this`，所以绑定失败
:::

## v-for 中 key 的作用

:::info[key 的作用]
key 是节点的唯一标识，帮助 Diff
算法`精准识别节点复用`，避免复用带来的渲染错误问题，同时提升更新效率。
:::

## watch vs watchEffect

1️⃣ 核心区别（极简版）

| 维度       | watch          | watchEffect               |
| ---------- | -------------- | ------------------------- |
| 依赖收集   | 显式指定监听源 | 自动收集                  |
| 默认执行   | 懒执行         | 立即执行                  |
| 拿到旧值   | ✅             | ❌                        |
| 控制精度   | 高             | 中                        |
| 异步稳定性 | 更稳           | 有陷阱（仅跟踪 await 前） |

2️⃣ `watchEffect` 高频追问速记

- 为何立即执行？ → 为了`首次执行 + 依赖收集`。
- 为什么触发过多？ → 回调中读到的`所有响应式数据`都会成为依赖。
- 异步陷阱？ → 只跟踪 `await 之前`的依赖。
- 连续请求污染咋办？ → 用 `onCleanup` 取消上一次的副作用。

3️⃣ 使用场景

- `watch`：需要旧值、明确指定监听源、复杂异步、延迟执行。
- `watchEffect`：依赖多、自动跟踪、执行时响应式数据用到谁就监听谁、初始化执行。

:::info[停止监听]
`watch` 和 `watchEffect` 都返回一个停止函数，调用返回函数即可停止监听；
在 `setup`/ `<script setup>` 中使用时，组件卸载会自动停止，通常无需手动处理。
:::

## computed vs watch

1️⃣ 核心区别（极简版）

| 维度   | computed                                   | watch          |
| ------ | ------------------------------------------ | -------------- |
| 定位   | 派生新值                                   | 执行副作用     |
| 返回值 | 有（计算结果）                             | 无（停止函数） |
| 缓存   | 有（依赖项变化重新计算，否是返回缓存结果） | 无             |
| 异步   | ❌ 不适合                                  | ✅ 适合        |
| 副作用 | ❌ 不应有                                  | ✅ 专为其设计  |

2️⃣ 使用场景

- `computed`：模板加工、组合状态、过滤排序、状态派生、结果计算、表单校验。
- `watch`：请求、缓存、路由联动、埋点、第三方实例操作。

3️⃣ `computed` 和 `watch` 都能监听数据变化，怎么选择？

- `computed`：用于`派生`数据，有返回值，可缓存计算结果。示例：fullName/totalPrice
- `watch`：用于`副作用`，无返回值。示例：发请求/操作 DOM/路由跳转

## computed vs methods

- `computed` 基于响应式依赖派生出新状态，只有依赖变化时才重新计算，否则使用缓存。
- `methods` 每次调用都会执行，无缓存机制。

📚 computed vs methods（核心对比）

| 对比维度         | computed                     | methods                    |
| :--------------- | :--------------------------- | :------------------------- |
| **缓存机制**     | ✅ 有缓存，依赖不变不重算    | ❌ 无缓存，每次调用都执行  |
| **触发时机**     | 依赖的响应式数据变化时       | 调用时立即执行             |
| **性能**         | 高（缓存减少无用计算）       | 低（频繁调用可能浪费性能） |
| **是否支持传参** | ❌ 不支持                    | ✅ 支持                    |
| **返回值**       | 必须返回一个值               | 可有可无                   |
| **使用场景**     | 过滤排序、状态派生、结果计算 | 事件处理/复杂逻辑/传参计算 |

## watch vs methods

- `watch`：数据变化时自动触发
- `methods`：每次调用都会触发

📚 watch vs methods（核心对比）

| 对比维度             | watch                       | methods                    |
| :------------------- | :-------------------------- | :------------------------- |
| **触发方式**         | 数据变化时自动触发          | 主动触发                   |
| **执行时机**         | 数据变化后立即执行          | 调用时立即执行             |
| **是否支持异步**     | ✅ 支持                     | ✅ 支持                    |
| **是否支持深度监听** | ✅ 支持 `deep: true`        | ❌ 无此概念                |
| **初始执行**         | ✅ 支持 `immediate: true`   | ❌ 无此概念                |
| **使用场景**         | 数据变化后的副作用/异步请求 | 事件处理/复用逻辑/普通函数 |

## computed/watch/methods

| 对比维度               | computed                  | watch               | methods           |
| :--------------------- | :------------------------ | :------------------ | :---------------- |
| **是支持缓存**         | ✅ 是                     | ❌ 否               | ❌ 否             |
| **触发方式**           | 自动（依赖变化）          | 自动（数据变化）    | 主动（调用）      |
| **执行时机**           | 读取时/依赖项目变化是执行 | 数据变化时执行      | 调用时执行        |
| **是否支持传参**       | ❌ 否                     | ❌ 否               | ✅ 是             |
| **是否必须返回值**     | ✅ 是                     | ❌ 否               | ❌ 否             |
| **是否适合异步**       | ❌ 否                     | ✅ 是               | ✅ 是             |
| **是否支持 deep**      | ✅ 是（访问深层时自动）   | ✅ 是（手动配置）   | ❌ 否             |
| **是否支持 immediate** | ❌ 否                     | ✅ 是               | ❌ 否             |
| **典型场景**           | 模板计算/派生状态         | 数据变化副作用/请求 | 事件处理/复用逻辑 |

:::warning[注意事项]
- `computed` 和 `method` 都能实现的功能，建议使用 `computed`，因为有缓存
- `computed` 和 `watch` 都能实现的功能，建议使用 `computed`，因为更简洁
:::

## 响应式原理

### 核心对比

| 对比维度   | Vue 2                                    | Vue 3                           |
| ---------- | ---------------------------------------- | ------------------------------- |
| 底层 API   | `Object.defineProperty`                  | `Proxy`                         |
| 劫持粒度   | 逐个 key                                 | 整个对象                        |
| 新增属性   | ❌ 需要 `$set`，因为初始化时没有这个 key | ✅ 自动检测                     |
| 删除属性   | ❌ 需要 `$delete`                        | ✅ 自动检测                     |
| 数组下标   | ❌ 需要 `$set`                           | ✅ 自动检测                     |
| 初始化性能 | 递归遍历所有 key，性能差                 | 惰性代理，访问时才递归，性能好  |
| 兼容性     | IE9+                                     | 不支持 IE (Proxy 无法 polyfill) |

### Object.defineProperty

- 初始化时遍历 `data` 的每个属性，添加 `getter/setter`
- **无法监听新增属性** → 需要 `Vue.set()`
- **无法监听 delete** → 需要 `Vue.delete()`
- **无法监听数组索引赋值** → 性能考虑，未对数组索引做劫持
- **无法监听数组 length 变化** → `length` 不是响应式属性
- vue2 重写了 `push`/`pop`/`shift`/`unshift`/`splice`/`sort`/`reverse` 七个方法

### Proxy

- 代理整个对象，拦截所有操作
- 自动监听新增属性和删除操作
- 惰性代理：访问到哪层才代理哪层
- 原生支持数组索引和 `length` 变化

### Vue 3 响应式流程

```text
数据读取 (get)
  ↓
track(target, key) — 收集 effect 到依赖 Map
  ↓
数据修改 (set)
  ↓
trigger(target, key) — 找到并执行依赖中的 effect
  ↓
组件重新渲染 / computed 重新计算 / watch 回调
```

## ref vs reactive

| 特性       | `ref`       | `reactive`         |
| ---------- | ----------- | ------------------ |
| 适用类型   | 基础 + 引用 | 仅对象             |
| 访问方式   | `.value`    | 直接访问属性       |
| 替换整体值 | 支持        | 不支持（丢失代理） |
| 解构       | 安全        | 丢失响应式         |

:::info
Vue 3.2+ 官方推荐**优先使用 ref**：统一 `.value`
语法，无解构丢失问题，适合整体替换。
:::

```ts
const state = reactive({ name: "Tom", age: 18 });
const { name, age } = state; // 丢失响应式
const { name, age } = toRefs(state); // 每个属性变成 ref，保持响应式
```

:::warning[ref 与 reactive 的解构]
- `reactive` 解构后丢失响应式！用 `toRefs()` 包装或改用 `ref`。 - Vue 3.5+
支持响应式 Props 解构，但仅限 `defineProps` 返回值，普通 `reactive` 仍需
`toRefs`。
:::

✅ ref 与 reactive 选择策略

```ts
// 基本类型 → 只能 ref
const count = ref(0)

// 对象且不需要整体替换 → reactive 更方便（不用 .value）
const form = reactive({ name: '', age: 0 })

// 对象且需要整体替换 → ref => (reactive 整体替换会丢失响应式)
const user = ref({ name: 'Tom' })
user.value = { name: 'Jerry' }  // ✅ ref 可以整体替换
```

## ref vs shallowRef

```ts tab="ref（深层）"
// ref — 深层响应，修改嵌套属性也触发更新
const user = ref({ name: "John", profile: { age: 30 } });
user.value.profile.age = 31; // 触发更新
```

```ts tab="shallowRef（浅层）"
// shallowRef — 只追踪 .value 赋值，不追踪嵌套变化
const data = shallowRef({ items: [] });
data.value.items.push("new"); // 不触发更新！
data.value = { items: ["new"] }; // 触发更新

// 大型不可变数据推荐用 shallowRef，避免深层代理开销
```

:::info[ref 与 shallowRef 选择策略]
- **优先 `shallowRef`** 用于大型数据结构或不需要深层响应的场景。 - `ref`
对基础类型用类包装（`.value` 的 get/set），对对象类型内部调用 `reactive`。
:::

## shallowRef & shallowReactive

| API               | 作用                                                   |
| ----------------- | ------------------------------------------------------ |
| `shallowRef`      | 只有 `.value` 这一层是响应式的，内部属性变化不触发更新 |
| `shallowReactive` | 只有对象的`第一层属性`是响应式的，深层嵌套对象不代理   |

## v-show vs v-if

| 维度     | `v-if`         | `v-show`        |
| -------- | -------------- | --------------- |
| 控制     | 添加/删除 DOM  | `display: none` |
| 初始渲染 | 条件为真才渲染 | 总是渲染        |
| 切换开销 | 高（重建 DOM） | 低（切换样式）  |
| 适用场景 | 条件很少变化   | 频繁切换        |
| 生命周期 | 触发           | 不触发          |

## v-if 与 v-for 优先级

| 版本  | 优先级         | 说明         |
| ----- | -------------- | ------------ |
| Vue 2 | `v-for > v-if` | 先遍历再判断 |
| Vue 3 | `v-if > v-for` | 先判断再遍历 |

:::warning
不要在同一元素上使用 `v-if` 和 `v-for`。用 `computed` 过滤列表或将 `v-if`
移到外层。
:::

## nextTick

### 概念

`nextTick` 是 Vue 提供的 `异步 DOM 更新机制`，**用于在下次 DOM 更新循环结束后执行回调**，确保能获取到更新后的 DOM。

```js
msg.value = "Vue3 NextTick!";
console.log(box.value?.textContent); // 可能是旧值

nextTick(() => {
  console.log(box.value?.textContent); // 确保拿到新值
});
```

### 使用场景

- 修改数据后读取更新后的 DOM
- `v-if` 创建元素后初始化第三方库
- 动态显示输入框后设置焦点

## 内置指令

### v-memo 与 v-once

:::info[内置指令]
- v-memo：通过`缓存`来跳过不必要的组件更新，从而优化性能。 -

```vue
<div v-memo="[]">永不更新</div>
```

## mixin

### mixin是什么

- `Mixin` 是面向对象程序设计语言中的类，提供了方法的实现。
- 其他类可以访问 `Mixin` 类的方法，而不必成为其子类。
- `Mixin` 类通常作为功能模块使用，在需要该功能时“混入”。
- `Mixin` 本质其实就是一个js对象，它可以包含我们组件中任意功能选项
  - data
  - components
  - methods
  - created
  - computed
  - ...

### 混入分类

- 局部混入
- 全局混入

### Vue的几种类型的合并策略

- 替换型：props、methods、inject、computed
  - 同名的props、methods、inject、computed会被后来者代替
- 合并型：data
  - 当目标 data 对象不包含当前属性时，调用 set 方法进行合并（set方法其实就是一些合并重新赋值的方法）
  - 当目标 data 对象包含当前属性并且当前值为纯对象时，递归合并当前对象值，这样做是为了防止对象存在新增属性

- 队列型：全部生命周期和watch
  - 生命周期钩子和watch被合并为一个数组，然后正序遍历一次执行
- 叠加型：component、directives、filters
  - 加型主要是通过原型链进行层层的叠加

小结 ✴️

- 替换型策略有：`props`、`methods`、`inject`、`computed`，就是将新的同名参数替代旧的参数
- 合并型策略是：`data`, 通过set方法进行合并和重新赋值
- 队列型策略有：`生命周期函数`和`watch`，原理是将函数存入一个数组，然后正序遍历依次执行
- 叠加型有：`component`、`directives`、`filters`，通过原型链进行层层的叠加

### 注意事项

- 混入存在相同选项
> 当组件存在与mixin对象相同的选项的时候，进行递归合并的时候`组件的选项`会`覆盖 mixin 的选项`
- 生命周期队列性合并
> 但是如果相同选项为生命周期钩子的时候，会合并成一个数组，先执行mixin的钩子，再执行组件的钩子

## 多次改变同一状态，是一次还是多次执行

- **同步连续修改**：批处理，一次更新 — 即一次执行
> Vue 会对同步的多次状态修改进行`批处理`，只触发一次组件重新渲染
- **异步修改**：多次更新（跨事件循环）— 即多处执行
> 若状态修改分散在不同的事件循环中，每次都会触发更新
- **computed/watch/watchEffect**：同步修改也只执行一次

## 常见误区

| 误区                          | 正解                                                            |
| ----------------------------- | --------------------------------------------------------------- |
| `reactive` 解构后还是响应式   | 解构丢失代理，用 `toRefs` 或改用 `ref`                          |
| `computed` 可以替代 `watch`   | `computed` 是纯派生，`watch` 是副作用                           |
| `nextTick` 一定是微任务       | 实现取决于环境（`Promise` → `MutationObserver` → `setTimeout`） |
| Vue 3 不需要 `key`            | 列表 `diff` 仍然需要 `key` 来识别节点                           |
| `v-if` 和 `v-for` 可以同元素  | Vue 3 优先级变了但仍然不推荐，用 `computed`                     |
| Pinia action 里不能异步       | Pinia action 天然支持 `async/await`                             |
| `shallowRef` 修改嵌套值能触发 | 只追踪 `.value` 赋值，嵌套修改需替换整个值                      |
