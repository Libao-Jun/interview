## HTTP 演进

### HTTP/1.1（1997）

- 默认 Keep-Alive 持久连接
- 同一连接串行请求 → **队头阻塞**
- 纯文本头部，重复传输
- 并发策略：浏览器对同域名开 6~8 个 TCP 连接

### HTTP/2（2015）

- **多路复用**：单连接并发多个流
- **二进制分帧**：HEADERS + DATA 帧
- **HPACK 头部压缩**：静态/动态表
- **服务端推送**（实际使用少）
- 局限：TCP 层队头阻塞仍存在

### HTTP/3（2022）

- 基于 **QUIC（UDP）**，不再用 TCP
- 彻底解决队头阻塞（每流独立）
- **0-RTT / 1-RTT** 握手（TLS 1.3 集成）
- **连接迁移**：Connection ID 标识，Wi-Fi 切 4G 不断连

### 对比总表

| 特性     | HTTP/1.1         | HTTP/2           | HTTP/3            |
| -------- | ---------------- | ---------------- | ----------------- |
| 传输协议 | TCP              | TCP              | QUIC (UDP)        |
| 队头阻塞 | 应用层 + 传输层  | 传输层 (TCP)     | 无                |
| 头部压缩 | 无               | HPACK            | QPACK             |
| 连接迁移 | 不支持           | 不支持           | 支持              |
| 建连延迟 | TCP 3-RTT + TLS  | TCP + TLS ≈ 2-3 RTT | 1-RTT / 0-RTT |

## OSI vs TCP/IP网络模型

| OSI七层网络模型         | TCP/IP四层概念模型 | 对应网络协议  |
| ----------------------- | -----------| --------------------------------------- |
| 应用层     | 应用层     | HTTP、TFTP、FTP、NFS、WAIS、SMTP        |
| 表示层     | 应用层     | Telnet、Rlogin、SNMP、Gopher            |
| 会话层     | 应用层     | SMTP、DNS                               |
| 传输层     | 传输层     | TCP、UDP                                |
| 网络层     | 网络层     | IP、ICMP、ARP、RARP、AKP、UUCP          |
| 数据链路层 | 数据链路层 | FDDI、Ethernet、Arpanet、PDN、SLIP、PPP |
| 物理层     | 数据链路层 | IEEE 802.1A、IEEE 802.2 到 IEEE 802.11  |

<Callout title="各层基本作用">
```txt
应用层 ---------> 为应用程序提供网络服务
表示层 ---------> 数据格式化、加密、解密
会话层 ---------> 建立、维护、管理会话连接
传输层 ---------> 建立、维护、管理端到端连接
网络层 ---------> IP寻址、路由选择
数据链路层 ---------> 控制网络层与物理层之间的通信；数据包（帧）
物理层 ---------> 光缆、电缆、无限电波；比特流（0｜1）传输
```
</Callout>

## HTTP 常用方法

1️⃣ 介绍

- get：获取资源，参数一般放 URL 中
- post：提交数据，常用于新增资源或执行操作
- put：整体更新资源
- patch：部分更新资源
- delete：删除资源
- options：查询服务器支持的请求方法，CORS 预检常用
- head：和 GET 类似，但只返回响应头，不返回响应体

2️⃣ 总结记忆口诀

```txt
GET 查数据不修改，参数都往 URL 放
POST 新增或提交，非幂等来记得牢
PUT 整体替换掉，幂等要求要知道
PATCH 部分改一改，不用传全套数据来
DELETE 删资源，幂等操作别心软
OPTIONS 问能力，CORS 预检就找它
HEAD 只取响应头，检查状态它最牛
```

## HTTP 状态码

[HTTP 响应状态码](https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Reference/Status)

1️⃣ 分类

- 1xx：信息响应（较少见）
- 2xx：成功响应
- 3xx：重定向，浏览器自动
- 4xx：客户端错误
- 5xx：服务器错误

2️⃣ 前端常见 HTTP 状态码

- 2xx 成功响应
  - 200：请求成功
- 3xx 重定向
  - 301：永久重定向，如网站域名迁移、HTTP → HTTPS
  - 302：临时重定向，如未登录跳转登录页、A/B 测试
  - 304 Not Modified：资源未修改（使用本地缓存），浏览器会发送请求但无响应体
- 4xx 客户端错误（前端锅 — 参数/路径/权限/登录等的问题）
  - 400 Bad Request：请求参数错误 — 检查请求参数/检查 Content-Type
  - 401 Unauthorized：未认证（未登录）— 检查是否有 token/检查 token 是否过期
  - 403 Forbidden：无权限（已登录但权限不足）— 检查用户角色/检查后端权限配置
  - 404 Not Found：资源不存在，API 路径写错/静态资源路径错误/SPA 刷新后 404（⭐ 前端最高频）
  - 405 Method Not Allowed：方法不允许
- 5xx 服务器错误（后端锅）
  - 500 Internal Server Error：服务器内部错误
  - 502 Bad Gateway：网关错误（后端服务挂了）
  - 503 Service Unavailable：服务维护/过载
  - 504 Gateway Timeout：网关超时（后端响应太慢）


## TCP vs UDP

<Callout type="info" title="TCP vs UDP">
TCP 是面向连接的可靠传输，UDP 是无连接的不可靠传输 — 一个重质量，一个重速度。
</Callout>

| 对比维度     | TCP                             | UDP                    |
| :----------- | :------------------------------ | :--------------------- |
| **连接性**   | 面向连接（三次握手四次挥手）    | 无连接（发完即走）     |
| **可靠性**   | 可靠，保证数据完整有序          | 不可靠，可能丢包乱序   |
| **速度**     | 慢（有流量控制、拥塞控制）      | 快（无额外机制）       |
| **数据边界** | 字节流，无边界                  | 数据报，有边界         |
| **适用场景** | (HTTP/HTTPS)/文件传输/WebSocket | 直播/视频通话/DNS/游戏 |

### 选择策略

- 数据重要 → TCP；丢了也行 → UDP
- 实时性高 → UDP；否则 → TCP

## 三次握手与四次挥手

### 三次握手（建立连接）

```
Client                          Server
  |── SYN (seq=x) ──────────────>|    我要连接
  |<── SYN+ACK (seq=y,ack=x+1) ──|    同意 + 确认
  |── ACK (ack=y+1) ────────────>|    确认，连接建立
```

<Callout>
**为什么不是两次？** 两次握手下，服务端无法确认客户端收到了 SYN+ACK，可能建立无效连接浪费资源。
</Callout>

### 四次挥手（断开连接）

```
Client                          Server
  |── FIN ──────────────────────>|    我不再发数据
  |<── ACK ──────────────────────|    收到（服务端可能还有数据）
  |<── FIN ──────────────────────|    我也不再发了
  |── ACK ──────────────────────>|    收到，进入 TIME_WAIT
```

<Callout>
**为什么不是三次？** TCP 全双工，双方需独立关闭。中间的 ACK 和 FIN 不能合并，因为服务端可能还有数据要发。
</Callout>

## HTTP 缓存

### 强缓存

不发请求，直接使用本地缓存。

| 头部              | 说明                         |
| ----------------- | ---------------------------- |
| `Cache-Control`   | `max-age=3600` / `no-cache` / `no-store` |
| `Expires`         | HTTP/1.0 绝对时间（已过时） |

### 协商缓存

发请求验证资源是否过期。

| 请求头              | 响应头            | 说明         |
| ------------------- | ----------------- | ------------ |
| `If-None-Match`     | `ETag`            | 内容哈希     |
| `If-Modified-Since` | `Last-Modified`   | 最后修改时间 |

### 缓存流程

```
请求资源 → 有缓存？
  ├─ 强缓存有效 → 直接使用
  └─ 强缓存过期 → 协商验证
       ├─ 304 → 使用旧缓存
       └─ 200 → 使用新资源 + 更新缓存
```

### 实战配置

```nginx
# HTML：每次验证
location / {
  add_header Cache-Control "no-cache, must-revalidate";
}

# 带 hash 静态资源：长期缓存
location /assets/ {
  add_header Cache-Control "public, max-age=31536000, immutable";
}
```

| 范围 | 类别     | 常见码                                             |
| ---- | -------- | -------------------------------------------------- |
| 2xx  | 成功     | `200 OK`                                           |
| 3xx  | 重定向   | `301` 永久 / `302` 临时 / `304` 缓存              |
| 4xx  | 客户端错 | `400` 参数错 / `401` 未认证 / `403` 无权限 / `404` 未找到 |

## GET vs POST

| 维度     | GET              | POST              |
| -------- | ---------------- | ----------------- |
| 语义     | 获取资源         | 提交数据          |
| 参数位置 | URL              | 请求体            |
| 幂等性   | 幂等             | 非幂等            |
| 缓存     | 可缓存           | 不缓存            |
| 安全性   | 参数暴露在 URL   | 相对更安全        |
| 长度限制 | URL 有长度限制   | 无限制            |

## WebSocket

| 特性     | HTTP             | WebSocket         |
| -------- | ---------------- | ----------------- |
| 通信模式 | 请求-响应        | 全双工            |
| 连接     | 短连接           | 持久连接          |
| 协议     | HTTP/1.1 / 2     | ws:// / wss://    |
| 握手     | —                | HTTP 101 升级     |
| 适用     | 普通请求         | 实时通信          |

## DNS 解析

1. 浏览器 DNS 缓存 → 系统缓存 → 路由器缓存
2. ISP 的 DNS 服务器 → 递归查询
3. 根 DNS → 顶级域 DNS → 权威 DNS

### 优化

```html
<!-- DNS 预解析 -->
<link rel="dns-prefetch" href="//api.example.com" />

<!-- 预连接（DNS + TCP + TLS） -->
<link rel="preconnect" href="https://api.example.com" />
```

## CDN

- 内容分发网络，将资源缓存到离用户最近的边缘节点
- 静态资源（JS/CSS/图片/字体）放 CDN
- 配合 `Cache-Control` 和文件名 hash 实现长期缓存 + 即时更新

## 网络安全

### HTTPS

- TLS 握手：身份认证 + 密钥协商 + 数据加密
- TLS 1.3：1-RTT 握手，0-RTT 重连

### 运营商劫持

- **DNS 劫持**：伪造 DNS 响应，指向广告页面
- **HTTP 劫持**：在响应中注入广告 JS/iframe
- **防御**：全站 HTTPS + HSTS + DNSSEC
