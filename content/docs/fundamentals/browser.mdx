## 从 URL 到页面渲染
```sh
1. DNS 解析：浏览器缓存 → 系统缓存 → 路由器缓存 → ISP DNS → 递归查询
2. TCP 连接：三次握手建立连接；HTTPS 还需 TLS 握手
3. HTTP 请求：发送请求报文，服务端返回响应
4. 响应处理：根据 `Content-Type` 决定处理方式（HTML / CSS / JS / 图片等）
5. 构建 DOM 树：HTML 解析 → 词法分析 → 构建节点树
6. 构建 CSSOM：CSS 解析 → 构建样式规则树
7. 构建渲染树：DOM + CSSOM → Render Tree（不含不可见元素）
8. 布局（Layout）：计算每个节点的几何信息（位置、大小）
9. 绘制（Paint）：将布局结果转换为像素
10. 合成（Composite）：GPU 合成各层，显示到屏幕
```

## HTTP 缓存策略

<Callout type="info" title="HTTP 缓存分类"> 
HTTP 缓存分为`强缓存`和`协商缓存`
 - 强缓存直接从浏览器缓存读取，不请求服务器；
 - 协商缓存向服务器发送请求，验证资源是否过期，由服务器决定是否使用缓存（304 使用缓存，200 + 资源更新）。
</Callout>

📚 核心对比

| 对比维度           | 强缓存                        | 协商缓存                                                          |
| :----------------- | :---------------------------- | :---------------------------------------------------------------- |
| **触发条件**       | 缓存有效期内                  | 缓存过期或用户强制刷新                                            |
| **是否请求服务器** | ❌ 否（直接读缓存）           | ✅ 是（验证资源是否变化）                                         |
| **响应状态码**     | 200（from disk/memory cache） | 304（Not Modified）                                               |
| **关键头部**       | `Cache-Control`、`Expires`    | `Last-Modified` / `If-Modified-Since`<br/>`ETag` / `If-None-Match` |
| **性能**           | 极快（无需网络请求）          | 较快（请求轻量，无 body）                                         |
| **控制方**         | 服务器通过响应头控制          | 服务器通过响应头和验证逻辑控制                                    |

2️⃣ 强缓存

```sh tab="Cache-Control"
Cache-Control: max-age=3600           # 缓存 1 小时
Cache-Control: no-cache               # 不使用强缓存，走协商缓存
Cache-Control: no-store               # 完全不缓存
Cache-Control: public                 # 允许所有中间节点缓存（CDN 等）
Cache-Control: private                # 仅浏览器缓存，不允许中间节点
Cache-Control: immutable              # 资源永不变化，可永久缓存
```
```sh tab="强缓存流程"
浏览器请求资源 → 检查 Cache-Control（max-age）
                    ↓
              是否在有效期内？
              ├─ 是 → 直接读缓存（200 from cache）✅
              └─ 否 → 发请求到服务器
```

3️⃣ 协商缓存

- 协商缓存流程（基于时间） — Last-Modified / If-Modified-Since
- 协商缓存流程（基于内容指纹，推荐） — ETag / If-None-Match

```sh tab="基于 Last-Modified"
服务器返回资源时带 Last-Modified（最后修改时间）
      ↓
再次请求时，浏览器携带 If-Modified-Since
      ↓
服务器对比时间
      ├─ 若资源未修改 → 返回 304，浏览器读缓存
      └─ 若资源已修改 → 返回 200 + 新资源
```
```sh tab="基于 ETag"
服务器返回资源时带 ETag（资源内容的唯一标识，如 Hash）
      ↓
再次请求时，浏览器携带 If-None-Match
      ↓
服务器对比 ETag
      ├─ 若一致 → 返回 304
      └─ 若不一致 → 返回 200 + 新资源
```

- Last-Modified 的缺陷

```sh
# 场景 1：秒级精度不足
文件在 14:20:30.500 修改 → Last-Modified: 14:20:30
14:20:30.800 再次修改 → Last-Modified 仍是 14:20:30 ❌ 检测不到变化

# 场景 2：时间篡改
用户手动修改系统时间 → Last-Modified 异常 ❌

# 场景 3：多服务器时间不同步
Server A: 14:20:30 | Server B: 14:20:35 → 同一文件 ETag 不同 ❌

# 场景 4：文件内容未变但时间变了
编辑后保存（内容不变）→ Last-Modified 变了 → 本应命中缓存却重新下载 ❌
```

🈴 什么场景适合强缓存？什么场景适合协商缓存？

- `强缓存`：不经常变化的资源，如`图片/字体/CSS/JS 库（CDN）`，设置长缓存（1 年）
- `协商缓存`：经常变化的资源，如 `HTML 入口文件/API 数据`，每次都验证

📚 ETag vs Last-Modified

- `ETag` 是基于内容指纹的精确验证，更可靠但性能开销稍大
- `Last-Modified` 是基于时间戳的粗略验证，更快速但存在精度缺陷

<Callout type="warn" title="混合使用"> 
两者配合使用，ETag 优先级更高。
</Callout>

| 对比维度         | ETag                               | Last-Modified                           |
| :--------------- | :--------------------------------- | :-------------------------------------- |
| **验证依据**     | 资源内容的唯一标识（Hash）         | 资源最后修改时间                        |
| **精度**         | ✅ 更高（内容变化即变化）          | ❌ 较低（秒级精度，毫秒级变化检测不到） |
| **性能开销**     | 稍大（需要计算 Hash）              | 较小（只读文件修改时间）                |
| **可靠性**       | ✅ 高（内容不变则不变）            | ❌ 低（时间可篡改、多服务器时间不同步） |
| **多服务器场景** | ✅ 一致（内容相同 ETag 相同）      | ❌ 不一致（文件时间可能不同）           |
| **优先级**       | ✅ **更高**（服务器优先比对 ETag） | ❌ 更低（ETag 存在时被忽略）            |
| **使用场景**     | 精确验证、CDN、分布式系统          | 静态文件、简单场景（降级方案）          |

<Callout type="info" title="为什么 ETag 比 Last-Modified 优先级更高？">
因为 ETag 更可靠
- `Last-Modified` 只能精确到秒，高并发场景下毫秒级变更检测不到
- `Last-Modified` 依赖文件系统时间，多服务器时间不同步会导致误判
- `Last-Modified` 文件内容未变但时间变了，编辑后保存（内容不变）→ Last-Modified 变了
- `ETag` 基于内容本身，不受时间和环境影响，准确性更高
</Callout>
- ETag 的 Weak 和 Strong 有什么区别？
  - `Strong ETag`："abc123"，内容必须`完全相同`才算匹配（强校验）
  - `Weak ETag`：W/"abc123"，允许内容`语义相同`（如字节不同但作用相同）即可匹配（弱校验，允许语义相同）

- ETag 有性能问题吗？如何优化？
  - ETag 需要计算 Hash，大文件有性能开销。优化方案如下
    - 对小文件（< 10MB）用完整 Hash
    - 对大文件用`文件大小 + 修改时间`组合生成 ETag（如 Nginx 默认 ETag: "size-mtime"）
    - 或用 CDN 服务商提供的 ETag 计算服务

- 两者同时配置，服务器决策（优先 ETag）
  - 1.先比较 `ETag` → 一致 ✅
  - 2.再比较 `Last-Modified` → 一致 ✅
  - 3.返回 `304 Not Modified`（浏览器使用缓存）


## 渲染管线

```
HTML ──→ DOM Tree ──┐
                     ├→ Render Tree → Layout → Paint → Composite
CSS ──→ CSSOM ──────┘
```

### 关键路径优化

- **减少关键资源数**：`async` / `defer` 加载 JS
- **减少关键路径长度**：内联关键 CSS
- **减少关键字节数**：压缩、Tree Shaking

### 阻塞行为

| 资源   | 阻塞解析 | 阻塞渲染 | 说明                     |
| ------ | -------- | -------- | ------------------------ |
| CSS    | 不阻塞   | 阻塞     | 构建 CSSOM 后才能渲染    |
| JS     | 阻塞     | 阻塞     | 默认阻塞 DOM 构建        |
| `async`| 不阻塞   | 可能     | 下载完立即执行，顺序不可控 |
| `defer`| 不阻塞   | 不阻塞   | DOM 解析后按序执行       |

## V8 引擎

### 执行流程

```
源代码 → 解析(Parser) → AST → 解释执行(Ignition) → 字节码
                                            ↓ 热点代码
                                    TurboFan 编译优化 → 机器码
                                            ↑ 优化假设失败
                                    Deoptimization 回退字节码
```

### 垃圾回收

| 区域   | 算法             | 对象类型     | 说明                   |
| ------ | ---------------- | ------------ | ---------------------- |
| 新生代 | Scavenge         | 朝生夕灭     | From → To 半区翻转     |
| 老生代 | Mark-Sweep/Compact | 长存活     | 标记清除 + 整理碎片    |

### 增量标记与并发回收

- **增量标记**：把长 GC 拆成短步，插入主线程间隙
- **并发回收**：GC 在辅助线程执行，不阻塞主线程
- **并行回收**：多个辅助线程同时 GC

## 跨域解决方案

同源策略：协议 + 域名 + 端口 三者一致

| 方案           | 原理                          | 特点                     |
| -------------- | ----------------------------- | ------------------------ |
| CORS           | 服务端设置响应头              | 最主流，需后端配合       |
| Nginx 代理     | 同源服务端转发                | 无需改后端，运维配置     |
| JSONP          | `<script>` 不受同源限制       | 只支持 GET，已过时       |
| WebSocket      | 协议本身不受同源限制          | 适合实时通信             |
| postMessage    | 跨窗口通信 API               | 适合 iframe / window     |

### CORS 关键头部

| 头部                          | 作用                     |
| ----------------------------- | ------------------------ |
| `Access-Control-Allow-Origin` | 允许的源                 |
| `Access-Control-Allow-Methods`| 允许的方法               |
| `Access-Control-Allow-Headers`| 允许的请求头             |
| `Access-Control-Allow-Credentials` | 是否允许 Cookie   |
| `Access-Control-Max-Age`      | 预检缓存时间             |

## Web 安全

### XSS（跨站脚本）

| 类型     | 攻击方式                 | 防御                         |
| -------- | ------------------------ | ---------------------------- |
| 存储型   | 恶意脚本存入数据库       | 输入过滤 + 输出转义         |
| 反射型   | URL 参数中注入脚本       | URL 编码 + CSP              |
| DOM 型   | 前端 JS 直接操作不可信数据 | 避免innerHTML / 使用安全API |

**核心防御**：输入验证 + 输出转义 + CSP（`Content-Security-Policy`）

### CSRF（跨站请求伪造）

| 防御方式       | 原理                         |
| -------------- | ---------------------------- |
| CSRF Token     | 服务端签发令牌，请求时携带验证 |
| SameSite Cookie| `Strict` / `Lax` 限制跨站携带 |
| Referer 校验   | 验证请求来源                 |

### 其他安全话题

- **点击劫持**：`X-Frame-Options: DENY` / CSP `frame-ancestors`
- **中间人攻击**：HTTPS + HSTS
- **SQL 注入**：参数化查询，不用字符串拼接

| ----------------- | ------- | ------------ | -------------- |

| localStorage      | 5MB+    | 永久         | 不携带         |

## Service Worker

- 独立线程，可拦截请求、管理缓存
- 生命周期：`install` → `activated` → `fetch`
- 核心应用：PWA 离线缓存、推送通知、后台同步
