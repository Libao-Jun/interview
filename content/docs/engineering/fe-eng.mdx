## 工程化演进 — 从手工到自动化

| 阶段 | 时代 | 代表工具 | 解决的问题 | 遗留问题 |
| ---- | ---- | -------- | ---------- | -------- |
| 1 | 手工时代 | 无 | — | 全局变量污染、手动管理依赖 |
| 2 | 任务自动化 | Grunt / Gulp | 重复任务自动化（压缩、合并、lint） | 没有模块系统，依赖手动管理 |
| 3 | 模块打包 | Webpack / Rollup | 模块化 + 按需加载 + Tree Shaking | 配置复杂，开发启动慢 |
| 4 | 零配置时代 | Vite / esbuild | 开发极速启动 + 开箱即用 | 生态迁移成本、插件兼容性 |

```
手工 (2005) → Grunt/Gulp (2012) → Webpack (2015) → Vite (2020)
手动管理   → 任务自动化       → 模块打包         → 零配置 + 原生 ESM
```

<Callout>
工程化演进的核心驱动力是**开发效率和交付质量的矛盾**。每一代工具都解决了上一代的痛点，但也引入了新的复杂度。理解这个演进逻辑，才能在面试中说清"为什么选这个工具"。
</Callout>

## 构建工具对比 — 选型的核心维度

| 维度 | Webpack | Vite | Rollup | esbuild |
| ---- | ------- | ---- | ------ | ------- |
| 开发启动 | 慢（全量打包） | 极快（原生 ESM，不打包） | — | — |
| HMR | 增量编译（随项目变大变慢） | 即时（模块级热替换） | — | — |
| 构建产物 | Chunk 分割灵活 | Rollup 输出 | Tree Shaking 优秀 | 极快但不优化 |
| 配置复杂度 | 高（概念多、插件链长） | 低（开箱即用） | 中 | 极低 |
| 适用场景 | 复杂应用、老项目 | 新项目首选 | 库和组件库 | 极速构建、转译 |

### Vite 架构 — 开发与构建的双模式

<Tabs items={['开发模式', '构建模式']}>

```
浏览器请求 → Vite Dev Server
                ↓
        拦截裸模块导入（如 'vue'）
                ↓
        按需编译单个模块（esbuild 预构建依赖）
                ↓
        返回原生 ESM → 浏览器直接执行

关键：不打包，浏览器原生 ESM 按需加载
     依赖预构建（esbuild）→ 首次启动后缓存
     源码按需编译 → 只编译当前页面用到的模块
```

```
Vite Build（生产构建）
        ↓
Rollup 打包 → Tree Shaking + 代码分割
        ↓
产物优化 → 压缩、CSS 提取、资源哈希
        ↓
输出静态资源（CDN 部署）

关键：生产用 Rollup（Tree Shaking 更成熟）
     开发用 esbuild（极快的按需编译）
     两套引擎，一套配置
```

</Tabs>

<Callout>
Vite 开发用 esbuild（编译快但不做优化），生产用 Rollup（Tree Shaking 成熟、产物优化好）。开发体验和生产产物由不同引擎负责，这是 Vite "快"的秘密——开发时不打包，只在请求时按需编译。
</Callout>

## 开发体验优化 — DX 也是生产力

### HMR — 模块热替换原理

| 维度 | Webpack HMR | Vite HMR |
| ---- | ----------- | -------- |
| 检测变更 | 文件系统监听 | 文件系统监听 |
| 处理方式 | 重新打包变更模块及其依赖 | 只重新编译变更模块 |
| 生效范围 | Chunk 级（可能影响多个模块） | 模块级（精确到单文件） |
| 速度 | 随项目变大变慢 | 始终快速（与项目规模无关） |

```
Webpack HMR 流程：
文件变更 → 重新编译 Chunk → WebSocket 通知浏览器 → 下载更新模块 → 替换模块

Vite HMR 流程：
文件变更 → 只编译变更模块 → WebSocket 通知浏览器 → 请求更新模块 → 替换模块
```

<Callout>
Vite HMR 速度与项目规模无关——因为它不需要重新打包，只编译变更的模块。Webpack HMR 需要重新编译整个 Chunk，项目越大越慢。
</Callout>

### Source Map — 定位源码的桥梁

| 配置 | 构建速度 | 适用场景 |
| ---- | -------- | -------- |
| `none` | 最快 | 生产环境（不暴露源码） |
| `eval` | 快 | 开发环境（快速重建） |
| `eval-source-map` | 中 | 开发环境（完整 Source Map） |
| `source-map` | 慢 | 生产环境调试（独立 .map 文件） |
| `hidden-source-map` | 慢 | 生产环境（不暴露 Source Map 引用，需手动上传到监控平台） |

<Callout>
生产环境推荐 `hidden-source-map`——不暴露 Source Map 引用给用户，但将 .map 文件上传到错误监控平台（Sentry），线上报错时可反定位到源码。
</Callout>

## 代码质量保障 — 自动化防线

### 四层防线 — 从编辑器到运行时

| 防线 | 时机 | 工具 | 作用 |
| ---- | ---- | ---- | ---- |
| 编辑器 | 编码时 | ESLint + Prettier 插件 | 即时提示错误和格式问题 |
| 提交门禁 | git commit 时 | husky + lint-staged | 只检查暂存区文件，阻止不规范代码入库 |
| CI 流水线 | push / PR 时 | GitHub Actions / CI | 全量检查 + 测试 + 构建 |
| 运行时监控 | 线上 | Sentry / 自建监控 | 捕获运行时错误，反馈到开发 |

### 提交门禁配置

```json
// package.json
{
  "lint-staged": {
    "*.{js,ts,vue}": ["eslint --fix", "prettier --write"],
    "*.{css,scss,vue}": ["stylelint --fix"],
    "*.{ts,vue}": ["vue-tsc --noEmit"]
  }
}
```

```bash
# .husky/pre-commit
npx lint-staged

# .husky/commit-msg
npx commitlint --edit $1
```

### 代码规范工具职责

| 工具 | 职责 | 关注点 |
| ---- | ---- | ------ |
| ESLint | 代码质量 | 逻辑错误、最佳实践、复杂度 |
| Prettier | 代码格式 | 缩进、换行、引号、分号 |
| TypeScript | 类型安全 | 类型错误、接口契约 |
| Stylelint | 样式规范 | CSS 属性顺序、选择器规范 |

<Callout>
ESLint 管逻辑，Prettier 管格式——职责不同，不要用 ESLint 的格式规则替代 Prettier。推荐 `eslint-config-prettier` 关闭 ESLint 中与 Prettier 冲突的规则。
</Callout>

## 环境与配置管理 — 多环境不失控

### .env 文件策略

| 文件 | 加载时机 | 用途 |
| ---- | -------- | ---- |
| `.env` | 所有环境 | 公共默认值 |
| `.env.local` | 所有环境（不提交 Git） | 个人本地覆盖 |
| `.env.development` | 开发环境 | 开发专属配置 |
| `.env.production` | 生产环境 | 生产专属配置 |
| `.env.staging` | 预发布环境 | 预发布专属配置 |

### 环境变量注入规则

```ts
// Vite — 只有 VITE_ 前缀的变量暴露给客户端
VITE_API_URL=https://api.example.com    // ✅ 暴露
SECRET_KEY=xxx                           // ❌ 不暴露

// 代码中访问
const apiUrl = import.meta.env.VITE_API_URL

// Webpack — 只有 REACT_APP_ / VUE_APP_ 前缀暴露
const apiUrl = process.env.REACT_APP_API_URL
```

| 原则 | 说明 | 反例 |
| ---- | ---- | ---- |
| 前缀过滤 | 只有特定前缀的变量暴露给客户端 | 所有环境变量都打包进前端 |
| 敏感信息不入库 | `.env.local` 加入 `.gitignore` | 密钥写进 `.env` 并提交 |
| 配置分离 | 业务配置与构建配置分开 | 业务 API 地址写在 vite.config.ts |
| 编译时替换 | 环境变量在构建时注入，不是运行时读取 | 前端代码 `process.env` 运行时读取 |

<Callout>
前端环境变量是**编译时替换**，不是运行时读取。`import.meta.env.VITE_API_URL` 在构建时被替换为字符串字面量。这意味着：构建后环境变量就固定了，切换环境必须重新构建。如果需要运行时切换配置，要用其他方案（如 `window.__CONFIG__` 注入）。
</Callout>

## 脚手架 — 项目初始化标准化

| 脚手架 | 创建命令 | 特点 |
| ------ | -------- | ---- |
| create-vue | `npm create vue@latest` | Vue 官方，支持 TS / Router / Pinia / ESLint 选项 |
| create-vite | `npm create vite@latest` | Vite 官方，多框架模板（Vue / React / Svelte） |
| create-react-app | `npx create-react-app` | React 官方（已不推荐，建议用 Vite 模板） |
| 自定义 CLI | 自建 | 统一团队模板、内嵌规范配置、一键创建 |

### 脚手架解决什么问题

| 问题 | 脚手架方案 |
| ---- | ---------- |
| 项目结构不统一 | 内置标准目录模板 |
| 依赖版本不一致 | 锁定核心依赖版本 |
| 规范配置遗漏 | 预装 ESLint / Prettier / husky / TS 配置 |
| 新项目搭建慢 | 一键创建，开箱即用 |

<Callout>
脚手架的价值不是生成代码，而是**固化团队的最佳实践**——目录结构、依赖版本、规范配置一次性到位，避免每个项目重新踩坑。团队规模越大，脚手架的收益越高。
</Callout>

## 工程化体系总结 — 从工具到流程

| 层级 | 关注点 | 关键工具/实践 |
| ---- | ------ | ------------- |
| 开发体验 | DX、HMR、类型提示 | Vite、TypeScript、ESLint 插件 |
| 代码规范 | 格式、质量、类型 | Prettier、ESLint、TS、commitlint |
| 提交门禁 | 阻止不规范代码入库 | husky + lint-staged |
| 构建优化 | 产物体积、构建速度 | Tree Shaking、代码分割、增量构建 |
| 环境管理 | 多环境配置隔离 | .env 策略、编译时替换 |
| CI/CD | 自动化构建和部署 | GitHub Actions、Turborepo |
| 运行时监控 | 线上错误捕获 | Sentry、Performance API |

<Callout>
工程化不是堆工具，而是**让每个环节都有自动化的保障**。工具选型的核心逻辑：能自动化的不手动、能在提交前发现的不留到 CI、能在 CI 发现的不留到线上。
</Callout>

### 编译或转译

1️⃣ 转译（Transpile）：特指将一种语言的源代码转换成另一种（或同一语言的不同版本）代码

- `.ts → .js`：TypeScript 转译为 JavaScript
- `.scss/.less → .css`：Sass/Less 转译为 CSS
- `现代 JS → 旧版 JS`：ES6+ 转译为 ES5

2️⃣ 预处理（Preprocessing）：主要针对 CSS 相关文件

- Sass/SCSS、Less、Stylus 被称为 CSS 预处理器，它们的`编译过程`叫 `预处理`

3️⃣ 降级编译：特指现代 JS → 兼容性代码

- 语法转换：`箭头函数 → function`，`const/let → var`
- API 补丁：通过 core-js 注入 Promise、Array.prototype.includes 等 polyfill
- Babel 称这个过程为 `preset-env` 转换

### Babel

<Callout title="定义">
`Babel` 是一个 `JavaScript 编译器`，主要用于将`新版本 ECMAScript 语法`转换为`向后兼容的旧版本语法`，
确保代码在老旧浏览器或环境中正常运行。
</Callout>

### 源码转换
- `React JSX` → 普通 `JS`
- `TS` → `JS`
- `ES6+` → `ES5`（新语法"翻译"成旧语法）
- 新提案语法 → 稳定语法
- 代码压缩/混淆
- 移除 `console.log` 等调试代码
- 确保浏览器兼容性

### 详细过程

- 1️⃣ 代码解析与依赖收集
  - Vite 从入口文件（通常是 `index.html` 和 `src/main.js`）开始，递归分析依赖（包括第三方依赖和本地模块）
  - 构建依赖图，确定哪些文件被使用

- 2️⃣ 模块转换
  - 将 `.vue` 文件拆解成` JS`、`CSS`、`HTML` 三部分
  - 将 `Vue` 模板编译成高效的 `render` 函数 — 通过 `@vitejs/plugin-vue` 插件
  - 转译：`.ts → .js`，`.scss/.less → .css`，`现代 JS（ES6+） → 兼容性代码（ES5）— 降级编译`

- 3️⃣ 代码优化
  - Tree Shaking：删除项目中未引用的代码（dead code）
  - 代码压缩：使用 `esbuild`/`terser` 移除空格、缩短变量名、简化逻辑
  - CSS 压缩：去除多余空格、注释
  - 图片优化：小图片可能转成 base64（根据 assetsInlineLimit 配置）

4️⃣ 代码分割
- 异步路由组件会生成独立的 chunk，实现按需加载

5️⃣ 文件命名与缓存策略
- 每个打包后的文件名会添加哈希值（如 index.a1b2c3d4.js）
- 文件内容变化时哈希改变，便于浏览器缓存和资源更新。

6️⃣ 生成构建产物（一般为dist）
- 输出到 `dist/` 目录，包含 HTML、JS、CSS、资源文件等。

### 图片资源处理

（一）若不配置插件扩展，对图片资源处理的规则是什么？

- 🗺️ 1.识别与收集：把你“用过”的图片找出来
- 🔀 2.决策与分流：决定每张图片的命运
- 👑 3.特殊处理：Public 目录 — 你说了算的“特权区”

（二）详解
1️⃣ 识别与收集图片资源 — 把你“用过”的图片找出来（Vite 扫描项目中所有被引用过的图片）

- 扫描范围：`import` 导入、CSS 的 `url()` 引用、`<img src="...">`标签
- 识别类型：默认情况下，常见的图片格式（如 PNG/JPG/PEG/SVG/GIF/WebP/AVIF 等）都会被自动识别并处理。
- 规则限制：只有被引用的图片才会被处理。（在`src/assets`目录下且`被引用的图片`才会被处理）

2️⃣ 决策与分流 — 决定每张图片的命运（Vite 会根据其`文件大小`，为它选择`两条`不同的处理路径，决策的临界点默认是 `4KB`—可通过`build.assetsInlineLimit` 配置项调整大小限制）

- 路径一：内联 (Inline) -> 转换为 Base64 格式
  - 条件 — 图片体积 `< 4KB`。
  - 处理方式 — Vite 会将这张小图片读取出来，转为 Base64，并直接“内嵌”到打包生成的 `JS/CSS` 文件里
  - 结果 — dist 文件夹里不会有一个单独的图片文件。（好处是减少请求次数，对于小图标能提升页面加载速度）
- 路径二：提取 (Extract) -> 变成独立文件并“加料”
  - 条件 — 图片体积 `>= 4KB`。
  - 处理方式 — Vite 会将这张图片当作文件处理
    - 复制并重命名：将图片复制到 `dist/assets` 目录下。
    - 添加哈希值：为了让浏览器更好地管理缓存，Vite 会为图片文件名添加一串哈希值。
  - 代码中替换路径：在生成的`JS/CSS`代码中，原来写图片路径的地方会被自动替换成这个新的、带哈希值的文件路径

3️⃣ 特殊处理：Public 目录

- 完全原样输出 — 放在 `public` 目录下的任何文件，在构建时会被`直接复制`到`dist`目录的根目录下，不做任何处理
- 没有体积检测，不会转 Base64。
- 不添加哈希值，文件名保持原样。
- 不会被 Tree Shaking 删除。
- 引用方式 — 必须使用`绝对路径`引用

（三）总结与建议

| 源位置             | 是否被引用 | 大小规则 | 最终处理方式                                 |
| ------------------ | ---------- | -------- | -------------------------------------------- |
| `src/assets/` 目录 | ✅         | `< 4KB`  | 转成 `Base64`，内嵌在 `JS/CSS` 中            |
| `src/assets/` 目录 | ✅         | `>= 4KB` | 输出到 `dist/assets/`，`文件名带哈希`        |
| `src/assets/` 目录 | ❌         | 任意     | 自动被忽略，不会被打包，不会出现在`dist`目录 |
| `public/` 目录     | 不检查     | 任意     | 原样复制到 dist/ 根目录，无哈希              |

#### 与开发模式的异同

| 特性     | `npm run dev`        | `npm run build`        |
| -------- | -------------------- | ---------------------- |
| 目标     | 快速开发调试         | 生产环境部署           |
| 底层依赖 | esbuild              | Rollup/Rolldown        |
| 输出     | 内存中，不写磁盘     | dist 文件夹            |
| 代码     | 未压缩、易调试       | 压缩、混淆、优化       |
| 模块     | 原生 ES 模块，未合并 | 合并、代码分割         |
| 特性     | 热更新、sourcemap    | Tree shaking、哈希命名 |

## Vite vs Webpack

1️⃣ 核心差异

| 维度       | Vite                                               | Webpack                                        |
| ---------- | -------------------------------------------------- | ---------------------------------------------- |
| 开发启动   | 原生 `ESModule` 按需加载，冷启动极快               | 先构建完整模块依赖图才能启动，项目越大越慢     |
| 依赖预构建 | `esbuild`（Go 编写），速度是 JS 打包器的 10-100 倍 | JS 实现的 `loader/plugin` 链，天然慢           |
| HMR 基于   | ESM，只更新变动模块，与项目规模无关                | 需要重新构建受影响的 `chunk`，项目大时明显变慢 |
| 生产构建   | `Rollup（Vite 5）/ Rolldown（Vite 6+）`            | 自身打包引擎                                   |
| 配置复杂度 | 开箱即用，约定优于配置                             | 灵活但配置繁重，学习曲线陡                     |

2️⃣ 为什么 Vite 开发阶段快

- 【不做全量打包】 — 浏览器直接请求 `ESM`，Vite 只在用到时按需编译单个模块
- 【依赖预构建用 esbuild】 — 把 `node_modules` 中的 `CJS/UMD` 转成 `ESM` 并合并小模块，只做一次
- 【HMR 精准】 — 模块变更只沿依赖链向上失效，不需要重建整个 bundle

## Vite vs Webpack构建过程

### Webpack 构建过程

```txt
源代码文件
    ↓
1. 解析入口（Entry）
    ↓
2. 递归解析所有 import/require，构建完整模块依赖图（Module Graph）
    ↓
3. 对每个模块执行 Loader 转换（babel、ts、scss、vue-loader 等）
    ↓
4. 应用 Plugin 处理（代码压缩、资源注入、环境变量替换等）
    ↓
5. 根据依赖图做代码分割（Code Splitting），生成 Chunk
    ↓
6. 输出最终 Bundle 文件（JS、CSS、图片等）
    ↓
浏览器加载 Bundle
```
<Callout type="warn" title="关键特征">
**先打包，再启动**。所有模块必须先经过完整编译和打包，才能提供给浏览器。
</Callout>

### Vite 构建过程（开发阶段）

```txt
源代码文件
    ↓
1. 启动开发服务器（几乎无需等待）
    ↓
2. 依赖预构建：用 esbuild 将 node_modules 中的 CJS/UMD 转为 ESM（仅首次或依赖变化时执行，结果缓存）
    ↓
3. 浏览器请求页面，加载入口 HTML
    ↓
4. 浏览器通过原生 ESM 的 import 逐个请求模块
    ↓
5. Vite 拦截请求，按需对单个模块做即时编译（TS、Vue SFC、SCSS 等）
    ↓
6. 编译后的 ESM 模块直接返回给浏览器
    ↓
浏览器原生执行 ESM
```

<Callout type="warn" title="关键特征">
**先启动，按需编译**。不构建完整 Bundle，浏览器请求哪个模块才编译哪个。
</Callout>

### Vite 构建过程（生产阶段）

```txt
源代码文件
    ↓
1. 用 Rollup（Vite 5）/ Rolldown（Vite 6+）做完整打包
    ↓
2. 构建模块依赖图
    ↓
3. Tree Shaking 移除未引用代码
    ↓
4. 代码分割、资源处理、压缩（移除未使用代码）
    ↓
5. 输出优化后的静态文件
    ↓
部署到 CDN / 服务器
```
<Callout type="warn" title="生产构建">
生产构建时 `Vite` 和 `Webpack` 流程类似，都需要完整打包。区别在于底层引擎不同。
</Callout>

### 核心差异对比

| 维度       | Webpack                     | Vite（开发）               | Vite（生产）         |
| ---------- | --------------------------- | -------------------------- | -------------------- |
| 启动方式   | 先打包，再启动              | 先启动，再按需编译         | 完整打包             |
| 模块处理   | 全部编译后合并为 Bundle     | 浏览器请求时单模块即时编译 | Rollup/Rolldown 打包 |
| 依赖处理   | 和业务代码一起打包          | esbuild 预构建 + 强缓存    | 同 Rollup 打包       |
| 浏览器加载 | 加载 Bundle 文件            | 原生 ESM 逐模块加载        | 加载 Bundle 文件     |
| 冷启动速度 | 慢（项目越大越慢）          | 快（与项目规模几乎无关）   | N/A                  |
| HMR 速度   | 重新构建受影响 chunk        | 仅失效变动模块，精准替换   | N/A                  |
| 转换工具   | Loader（babel 等，JS 实现） | esbuild（Go 实现）+ 插件   | Rollup 插件          |

### 为什么开发体验差距大

```txt
Webpack 开发启动：
  入口 → 递归解析所有模块 → 全部 Loader 转换 → 生成 Bundle → 启动服务器 → 浏览器加载
  [────────── 项目越大，这段越长 ──────────]

Vite 开发启动：
  启动服务器（毫秒级）→ 预构建依赖（esbuild，秒级，有缓存）→ 浏览器按需请求模块 → 即时编译返回
  [── 固定开销 ──]                                          [── 按需，首屏才编译 ──]
```

### 面试标准回答

- Webpack 的构建过程是**先打包再启动**：从入口递归构建完整依赖图，每个模块经过 Loader 转换，最终合并为 Bundle 文件交给浏览器。
- Vite 开发阶段则是**先启动再按需编译**
  - 利用浏览器原生 ESM，启动时几乎零等待，浏览器请求哪个模块才即时编译哪个；
  - 依赖预构建交给 esbuild 处理并缓存，所以冷启动和 HMR 都极快。
  - 生产阶段 Vite 仍需完整打包（用 Rollup/Rolldown），流程与 Webpack 类似。
- 核心区别在于`开发阶段`的架构思路不同：
  - Webpack 是 "编译时" 模型。
  - Vite 是 "运行时按需" 模型。

## Tree Shaking

0️⃣ 参考资料
<Callout type="warn" title="摇树优化">
[摇树优化](https://developer.mozilla.org/zh-CN/docs/Glossary/Tree_shaking): 指描述 JavaScript 上下文中`移除未引用代码（dead code）`行为的术语。
</Callout>

1️⃣ 定义

- 一种基于 `ES Module 的静态结构`，在打包阶段`移除未被引用代码`的`优化技术`。(名字很形象：在整个依赖树中将枯枝落叶 “摇晃” 掉，只留下活代码)

2️⃣ 工作流程（四步）

- 第 1 步：静态分析（构建依赖图）
  - 打包工具从入口文件开始，追踪所有 `import/export` 语句，构建出`模块依赖图`。
- 第 2 步：标记（Mark）
  - 遍历依赖图，从入口出发，标记所有被引用的导出。未被标记的就是`死代码`。
- 第 3 步：删除（Sweep）
  - 删除 `死代码` 对应的导出语句及内部实现。
- 第 4 步：压缩（Terser/UglifyJS），代码压缩工具进一步清理
  - 被删除导出后的空模块
  - 未被调用的函数
  - 无法到达的代码分支

3️⃣ Example

```js [静态分析,构建依赖图]
// math.js
export function add(a, b) {
  return a + b;
}
export function sub(a, b) {
  return a - b;
}

// index.js
import { add } from "./math"; // 只引入了 add
console.log(add(1, 2));
```

```txt [标记（Mark）]
入口 → index.js → 使用了 math.add，则标记为 可达代码
                → math.sub 未被引用，则为 死代码
# 完整流程
第1步：标记所有代码为"存活"（Live）
入口 → index.js → 所有导入的模块 → 所有导出
    → math.add ← 被 index.js 引用，标记为 Live
    → math.sub ← 虽然未被引用，但因为被 math 模块导出，暂标记为 Live

第2步：从入口开始，可达性分析（Reachability Analysis）
入口 → index.js → import { add } from './math'
                → 只有 add 被真正使用
                → sub 在这个引用链上不可达

第3步：消除不可达代码（Dead Code Elimination）
math.sub ← 不可达，移除！
```

```js [删除（Sweep）]
// math.js 打包后变成
function add(a, b) {
  return a + b;
}
// sub 函数被完全删除
```

`js [压缩（Terser/UglifyJS）]`

4️⃣ Tree Shaking 的能力边界

```txt
未引用代码（Tree Shaking 能处理）→ 标记哪些导出被引用了，删除未被引用的 export
  ├── 模块导出了但没人 import 的函数
  └── 被标记为无副作用的未引用导出

真正的无用代码/死代码（Tree Shaking 处理不了）
  ├── 未赋值变量：let a = 1; a 之后没被用过
  ├── 永远为 false 的 if 分支
  ├── 无法到达的代码：return 之后的代码
  ├── 声明但未调用的函数（内部函数）
  └── 空函数

删除无用代码/死代码（Terser 压缩）
   ├── 扫描每个模块的 AST
   ├── 删除无法到达的代码（return 后）
   ├── 删除未使用的局部变量（let a = 1; a 之后没被用过）
   ├── 删除空函数
   └── 删除 if(false) 分支
```

5️⃣ 注意事项⚠️

- Tree Shaking 删除的是 `未引用代码（unused exports）`，而不是无用代码/死代码。
- Tree Shaking 只处理 JS 模块，`不会清理未使用的 CSS`。

## 代码执行过程

<Callout title="完整流程">
`读取源码` → `词法分析` → `语法分析` → `生成AST树` → `编译为字节码` → `JIT 优化为机器码` → `创建执行上下文（变量提升/TDZ/this/作用域链）` → `执行（调用栈 + 事件循环调度宏/微任务）` → `垃圾回收（GC）`
</Callout>

### 总览流程图

```txt
① 读取源码（字符流）
        ↓
② 词法分析（Lexing / Tokenizing）
        ↓
   Token 流（关键字 / 标识符 / 运算符 / 字面量 …）
        ↓
③ 语法分析（Parsing）           ── 语法错误在此抛出 SyntaxError，整个脚本不执行
        ↓
   AST（抽象语法树）
        ↓
④ 编译（V8）
   ├── Ignition 解释器：AST → 字节码（Bytecode），逐条解释执行
   └── TurboFan JIT：热点字节码 → 优化机器码（类型假设失效则去优化回退到字节码）
        ↓
⑤ 创建执行上下文 EC
   ├── 变量环境：var / function 声明提升
   ├── 词法环境：let / const / class（进入 TDZ）
   ├── this 绑定
   └── 外部引用 → 形成作用域链
        ↓
⑥ 执行阶段（单线程，主线程 Event Loop 驱动）
   ├── 同步代码：函数调用 push 调用栈，return 时 pop（LIFO）
   ├── Web API：定时器 / I/O / 事件 由宿主线程处理，回调按类型入队
   │     ├── 微任务队列：Promise.then / queueMicrotask / MutationObserver
   │     └── 宏任务队列：setTimeout / setInterval / I/O / UI 事件 / MessageChannel
   └── 事件循环（每一轮 tick）：
         调用栈清空 → 一次性清空所有微任务 → 取 1 个宏任务执行
         → 再次清空所有微任务 → （浏览器：必要时渲染） → 下一轮
        ↓
⑦ 垃圾回收（GC，与执行并发/穿插进行）
   ├── 新生代：Scavenge（From / To 空间复制，空间换时间）
   └── 老生代：Mark-Sweep（标记清除）+ Mark-Compact（整理碎片）
```

### 面试一句话回答

<Callout title="JS 代码执行分五步">
 - 解析 — 读取源码 → 词法分析拆 Token → 语法分析 → 生成AST树）
 - 编译 — 编译为字节码（热点代码 JIT 为机器码）
 - 创建执行上下文 — 完成变量提升和作用域链构建
 - 执行 — 时通过调用栈管理函数调用、通过事件循环调度异步任务（微任务优先于宏任务）
 - 垃圾回收 — 回收无引用内存。
</Callout>

## 循环依赖

1️⃣ 定义

<Callout title="循环依赖的定义">
模块 A 依赖了模块 B，模块 B 又直接或间接依赖了模块 A，形成闭环，导致`模块无法正确初始化`。
</Callout>

2️⃣ 不同模块系统表现

```txt
循环依赖就是 A 导入了 B，B 又导入了 A，形成闭环。

CommonJS 能"容忍"循环依赖：遇到正在加载的模块，直接返回当前已导出的部分，
但这会导致可能拿到 undefined，造成难以排查的 bug。

ES Module 则更严格：静态分析时就能发现循环依赖，
执行时未初始化的绑定会直接抛出 ReferenceError。

解决方式：
1. 提取公共模块，把共同依赖的逻辑提取出来
2. 在 Vue 组件场景用 defineAsyncComponent 异步加载
3. 在 Pinia/Vuex 的 store 中不要顶层互相 import，而是放在 getter/action 内部延迟调用
4. 用 dpdm、madge 或 circular-dependency-plugin 自动检测

根本原则：模块设计应单向依赖，一旦出现循环就说明设计需要重构。
```

3️⃣ 解决方案总结

| 方案         | 适用场景          | 做法                                           |
| ------------ | ----------------- | ---------------------------------------------- |
| 提取公共模块 | 两个模块共享逻辑  | 把共同依赖的部分抽到第三个文件                 |
| 依赖反转     | A 依赖 B 的实例   | 通过事件/回调/接口解耦，不直接 `import`        |
| 动态导入     | 组件/模块循环引用 | `import()` 异步加载，打破静态依赖链            |
| 延迟调用     | `Store` 循环依赖  | 不要在顶层调用，放在函数内部按需调用           |
| 合并模块     | 两个文件高度内聚  | 直接放一个文件里，消除依赖                     |
| 依赖注入     | 复杂场景          | 通过 `provide/inject` 或参数传递，而非直接导入 |

## 幽灵依赖

1️⃣ 定义

<Callout title="定义">
代码中使用了某个包，但 `package.json` 里没有声明它。只是因为 `node_modules` 提升或其他依赖间接安装，导致当前环境暂时还能用。
</Callout>

2️⃣ 检测幽灵依赖

```sh
# 使用 depcheck
npx depcheck

# 输出示例：
# Unused dependencies
# * lodash          ← 你没声明，但实际在用

# Missing dependencies
# * dayjs           ← 你声明了，但代码里没用到
```

3️⃣ 预防幽灵依赖的工具

```sh
# eslint-plugin-import 检查导入
npm install eslint-plugin-import --save-dev

# .eslintrc.js
# module.exports = {
#   plugins: ['import'],
#   rules: {
#     'import/no-extraneous-dependencies': 'error',  // 禁止导入未声明的包
#     'import/no-unresolved': 'error',               // 禁止导入不存在的包
#   },
# };
```

4️⃣ 使用 pnpm 从根源消灭

```sh
# pnpm 天然杜绝幽灵依赖
pnpm install
# 如果代码里用了未声明的包 → 直接报错
```

5️⃣ 总结

```txt
幽灵依赖 = 正常跑了不该跑的依赖

原因：npm/yarn 把子依赖提升到顶层 node_modules，
      Node.js 递归向上查找模块时刚好能找到。

后果：本地能跑，生产/CI 可能崩，依赖版本不受控。

解决：pnpm（非扁平化）、eslint 规则检查、depcheck 扫描。
```

## 打包生成文件中hash的作用

1️⃣ 作用：主要用于`浏览器缓存管理`

- 确保文件内容变化时，浏览器能获取最新版本；内容未变时，充分利用缓存；
- 解决同名文件覆盖问题，使用 hash 后，旧版本文件保留，新版本另存为新文件，便于灰度发布或快速回滚。

2️⃣ 三种常见 Hash 策略

| 类型        | 说明                                              | 特点                                                |
| ----------- | ------------------------------------------------- | --------------------------------------------------- |
| hash        | 整个项目的所有文件共享同一个 hash                 | 任何一个文件改动，所有文件 hash 都变 → 缓存利用率低 |
| chunkhash   | 同一个入口（及其依赖）打包成一个 chunk，共享 hash | 仅该入口内的文件改动才影响此 chunk                  |
| contenthash | 只根据`当前文件自身内容`生成 hash                 | 最精确，适合 CSS 等独立文件                         |

<Callout title="总结">
`hash` 是实现前端 `长效缓存 + 精确更新` 的标准手段。
</Callout>

## 静态资源的处理

1️⃣ 处理方式

- `public`：直接复制，不经过打包工具处理
- `static`：直接复制，不经过打包工具处理
- `assets`：经过打包工具处理

2️⃣ 关键对比表

| 特性              | public       | assets          | static (CLI 2.x)    |
| ----------------- | ------------ | --------------- | ------------------- |
| 是否打包处理      | ❌           | ✅              | ❌                  |
| 文件名是否加 hash | ❌           | ✅              | ❌                  |
| 图片转 base64     | ❌           | ✅ (小于 limit) | ❌                  |
| 引用方式          | 绝对路径 `/` | 相对路径或 `@/` | 绝对路径 `/static/` |
| 是否经过依赖图    | ❌           | ✅              | ❌                  |
| 热更新            | ❌           | ✅              | ❌                  |

3️⃣ 项目中静态资源分类建议

```js
✅ 会变动、需要优化的 → src/assets
  - UI 图标
  - 背景图
  - 组件内图片
  - ...

✅ 绝对不变、需要保持原名 → public
  - favicon.ico
  - 第三方库文件（如 Swiper 的字体）
  - ...

❌ 不要用 static（除非维护老项目）
```

## 常见误区

| 误区 | 正解 |
| ---- | ---- |
| 工程化就是配置 Webpack | 工程化是从开发到交付的完整流程，Webpack 只是构建环节的工具 |
| Vite 生产构建也用 esbuild | Vite 生产构建用 Rollup，esbuild 只用于开发时的按需编译和依赖预构建 |
| ESLint 能替代 Prettier | ESLint 管逻辑质量，Prettier 管代码格式，职责不同，需配合使用 |
| 环境变量是运行时读取 | 前端环境变量是编译时替换，构建后固定，切换环境需重新构建 |
| Source Map 生产环境不能开 | 用 `hidden-source-map` + 上传监控平台，不暴露给用户但可反定位源码 |
| lint-staged 检查全量代码 | lint-staged 只检查暂存区文件，速度远快于全量 lint |
| 脚手架只是生成模板 | 脚手架的核心价值是固化团队最佳实践，统一规范和依赖版本 |
